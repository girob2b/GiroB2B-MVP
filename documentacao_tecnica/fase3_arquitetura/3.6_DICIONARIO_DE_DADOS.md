Ref: ERD 2.5 §1.1, Padrões 3.4 §6.1.

| Elemento | Padrão | Exemplo |

### 0.3 Padrões de Audit Fields

Ref: ERD 2.5 §5, Padrões 3.4 §6.2.

| Padrão | Campos | Aplicado em |
|--------|--------|-------------|
| **A** — Ciclo de vida completo | `created_at`, `updated_at`, `deleted_at` | `profiles`, `suppliers`, `products` |
| **B** — Mutável sem soft delete | `created_at`, `updated_at` | `buyers`, `categories`, `plans`, `subscriptions`, `leads`, `notification_preferences` |
| **C** — Imutável (log/evento) | `created_at` | `supplier_categories`, `product_images`, `supplier_images`, `inquiry_responses`, `distribution_rounds`, `credits_weekly`, `credits_extra`, `credit_transactions`, `payment_attempts`, `reports`, `admin_actions`, `verification_documents`, `notifications`, `email_logs`, `search_logs`, `favorites`, `buyer_alerts`, `locations`, `product_imports` |
| **D** — Configuração | `updated_at`, `updated_by` | `system_configs` |

Trigger `update_updated_at()` aplicado automaticamente em todas as tabelas com Padrão A ou B (ver Seção 4).

### 0.4 Estratégia de Soft Delete

- Implementado via `deleted_at TIMESTAMPTZ` (nullable) — nunca `is_deleted BOOLEAN` (DM-06)
- Apenas 3 tabelas usam soft delete: `profiles`, `suppliers`, `products`
- Janela de recuperação: **30 dias** (RN-02.05)
- Job agendado faz hard delete após expiração da janela
- Todas as queries de leitura pública filtram `WHERE deleted_at IS NULL`
- Após hard delete de `suppliers`, dados são anonimizados (LGPD): nome→hash, CNPJ→null, emails→null

### 0.5 Formato dos exemplos neste documento

- **UUIDs:** abreviados como `550e8400-...` (na implementação real: UUID v7 via `gen_random_uuid()`)
- **CNPJs:** formato 14 dígitos sem pontuação (ex: `12345678000190`), sempre fictícios
- **Datas:** ISO 8601 com timezone BRT (`2026-04-05T14:30:00-03:00`)
- **Valores monetários:** `DECIMAL(10,2)` em BRL (ex: `79.00`)
- **JSONB:** representado como JSON inline (ex: `{"mon-fri": "08:00-18:00"}`)

### 2.01 `profiles`

**Descrição:** Estende `auth.users` do Supabase. Tabela-ponte entre autenticação gerenciada e entidades de domínio. Todo usuário autenticado tem exatamente 1 profile. Representa o Nível 1 do cadastro unificado.
**Fase:** MVP
**Audit Pattern:** A (created_at, updated_at, deleted_at)
**RLS:** Usuário lê/edita apenas seu profile. Admin lê todos.

#### Colunas

| # | Coluna | Tipo | Null | Default | Constraint | Descrição | Exemplo | RNs |
|---|--------|------|------|---------|------------|-----------|---------|-----|

| 2 | `display_name` | `VARCHAR(255)` | YES | `NULL` | — | Nome de exibição do usuário | `João Silva` | — |

### 2.02 `suppliers`

**Descrição:** Dados cadastrais e perfil da empresa fornecedora. Entidade central do marketplace. Nível 3 do cadastro unificado. Perfil público combina esta tabela com `products`, `supplier_images` e `supplier_categories`.

## 5. Políticas RLS Detalhadas

### 5.1 Padrão geral

O Supabase RLS usa `auth.uid()` para identificar o usuário. Admin bypass via `service_role key` (Edge Functions) que ignora RLS completamente.

```sql
-- Padrão: habilitar RLS em toda tabela
ALTER TABLE {tabela} ENABLE ROW LEVEL SECURITY;
```

### 5.2 Policies por tabela

| Tabela | Policy | Operação | USING / WITH CHECK | Descrição |
|--------|--------|----------|-------------------|-----------|
| `profiles` | `profiles_select_own` | SELECT | `id = auth.uid()` | Lê próprio profile |
| `profiles` | `profiles_update_own` | UPDATE | `id = auth.uid()` | Edita próprio profile |
| `profiles` | `profiles_admin_all` | ALL | `EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin')` | Admin total |
| `suppliers` | `suppliers_select_public` | SELECT | `status = 'active' AND deleted_at IS NULL` | Perfil público |
| `suppliers` | `suppliers_select_own` | SELECT | `user_id = auth.uid()` | Lê próprio (completo) |
| `suppliers` | `suppliers_update_own` | UPDATE | `user_id = auth.uid()` | Edita próprio |
| `suppliers` | `suppliers_admin_all` | ALL | Admin check | Admin total |
| `buyers` | `buyers_select_own` | SELECT | `user_id = auth.uid()` | Lê próprio |
| `buyers` | `buyers_update_own` | UPDATE | `user_id = auth.uid()` | Edita próprio |
| `locations` | `locations_select_public` | SELECT | `true` | Público |
| `locations` | `locations_admin_write` | INSERT/UPDATE/DELETE | Admin check | Apenas admin escreve |
| `categories` | `categories_select_public` | SELECT | `true` | Público |
| `categories` | `categories_admin_write` | INSERT/UPDATE/DELETE | Admin check | Apenas admin escreve |
| `products` | `products_select_active` | SELECT | `status = 'active' AND deleted_at IS NULL` | Produtos públicos |
| `products` | `products_select_own` | SELECT | `supplier_id IN (SELECT id FROM suppliers WHERE user_id = auth.uid())` | Supplier vê todos os seus |
| `products` | `products_write_own` | INSERT/UPDATE/DELETE | Via supplier_id do owner | Supplier escreve os seus |
| `inquiries` | `inquiries_select_buyer` | SELECT | `buyer_id IN (SELECT id FROM buyers WHERE user_id = auth.uid())` | Buyer vê suas inquiries |
| `inquiries` | `inquiries_select_supplier` | SELECT | `supplier_id IN (SELECT id FROM suppliers WHERE user_id = auth.uid())` | Supplier vê endereçadas a ele |
| `reports` | `reports_insert_auth` | INSERT | `auth.uid() IS NOT NULL` | Qualquer autenticado denuncia |
| `reports` | `reports_select_own` | SELECT | `reporter_id = auth.uid()` | Vê próprias denúncias |
| `admin_actions` | `admin_actions_admin_only` | ALL | Admin check | Apenas admin |
| `notifications` | `notifications_select_own` | SELECT | `recipient_id = auth.uid()` | Lê próprias notificações |
| `notifications` | `notifications_update_read` | UPDATE | `recipient_id = auth.uid()` | Marca como lida |
| `system_configs` | `system_configs_select_public` | SELECT | `true` | Configs públicas |
| `system_configs` | `system_configs_admin_write` | UPDATE | Admin check | Apenas admin altera |

> **Admin bypass:** Edge Functions que operam como sistema (jobs agendados, distribuição, cobrança) usam `service_role key` do Supabase, que bypassa RLS completamente.
