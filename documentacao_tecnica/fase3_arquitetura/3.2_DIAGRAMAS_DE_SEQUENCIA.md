## Resumo Executivo

Este artefato documenta os **19 fluxos de interação detalhados** entre componentes, serviços externos e atores para todos os cenários críticos do GiroB2B. Enquanto o artefato 3.1 (Diagrama de Componentes) mostra a visão **estática** — quais componentes existem e como se organizam —, este artefato complementa com a visão **dinâmica**: como os componentes interagem ao longo do tempo em cada cenário.

Os 19 cenários (SEQ-01 a SEQ-19) estão organizados em 7 grupos temáticos e cobrem **23 dos 33 Use Cases** documentados no 2.1 (v1.1). Os 10 UCs não cobertos são operações CRUD simples ou painéis admin sem interação cross-domain significativa (justificativa na seção 9). Cada diagrama inclui o fluxo principal, fluxos alternativos (`alt`/`opt`), tratamento de erros (`break`) e notas de implementação para o dev.

**Nota sobre o 3.1:** O artefato 3.1 (seção 8) contém 3 `sequenceDiagram` resumidos para ilustrar a comunicação entre componentes: Busca (8.1), Inquiry (8.2) e Desbloqueio de Lead (8.3). Este artefato **não duplica** esses diagramas — os **expande** com cenários completos, fluxos alternativos, tratamento de erros e edge cases. Os diagramas do 3.1 são referenciados como "versão simplificada" nos SEQs correspondentes (SEQ-06, SEQ-07, SEQ-11).

## 3. Grupo 1 — Identidade e Onboarding

### 3.1. SEQ-01: Cadastro de Usuário (Nível 1)

**Cenário:** Novo usuário cria conta genérica na plataforma. Não há escolha de role (comprador/fornecedor). Apenas dados básicos: email, senha, nome, telefone, cidade e estado. Resultado: conta Nível 1 que pode navegar e buscar, mas não pode enviar inquiries (precisa UC-12) nem listar produtos (precisa UC-31/SEQ-17).

**Pré-condições:**
- Nenhuma (qualquer pessoa pode se cadastrar)
- Usuário não possui conta ativa no GiroB2B

**Participantes:**

| Componente | Domínio | Estereótipo |
|-----------|---------|-------------|
| authService | Identity | `<<service>>` |
| profileRepository | Identity | `<<repository>>` |
| zodValidation | Cross-cutting | `<<middleware>>` |
| Supabase Auth | External | `<<external>>` |
| notificationService | Notifications | `<<service>>` |
| Resend | External | `<<external>>` |

**Diagrama:**

```mermaid
sequenceDiagram
    autonumber
    participant B as Browser
    participant API as API Register
    participant ZV as zodValidation
    participant AS as authService
    participant SA as Supabase Auth
    participant PR as profileRepo
    participant NS as notificationService
    participant RE as Resend

    B->>API: POST /api/auth/register
    Note right of B: email, password, name,<br/>phone, city, state
    API->>ZV: validate(createUserSchema)
    break Validação falha (campos obrigatórios, formato email)
        ZV-->>B: 400 Bad Request + detalhes Zod
    end

    API->>AS: registerUser(data)
    AS->>SA: createUser(email, password)
    break Email duplicado
        SA-->>AS: auth error
        AS-->>B: 409 Email já cadastrado
        Note over AS: Sugerir login ou recuperar senha
    end

    SA-->>AS: userId + session

    AS->>PR: create(profile: userId, name, phone, city, state)
    Note over PR: Cria em profiles apenas.<br/>SEM registro em buyers nem suppliers.

    par Envio de confirmação
        AS->>NS: send(userId, WELCOME_USER)
        NS->>RE: sendEmail(template: welcome + confirm link)
    end

    AS-->>B: 201 Created (profile, session)
    Note over B: RN-01.03 — 48h para confirmar,<br/>lembrete se não confirmar,<br/>7 dias para auto-remoção
```

**Fluxos alternativos:**

| # | Condição | Resultado | RN/UC |
|---|----------|-----------|-------|
| ALT-1 | Email já cadastrado | 409 — sugestão de login ou recuperar senha | EX-01.1 |
| ALT-2 | Senha não atende requisitos mínimos | 400 — critérios exibidos (8 chars, 1 maiúscula, 1 número) | EX-01.2 |
| ALT-3 | Validação Zod falha | 400 — campos inválidos detalhados | — |
| ALT-4 | Não confirma email em 48h | Lembrete enviado. 7 dias → conta auto-removida | FA-01.1, RN-01.03 |

**Pós-condições:**
- Conta criada no Supabase Auth **sem role específico** (Nível 1 — Usuário)
- Registro em `profiles` com dados básicos (nome, telefone, cidade, estado)
- **SEM registro em `buyers` nem `suppliers`**
- Sem CNPJ, sem dados de empresa, sem consentimento LGPD neste momento
- Usuário pode navegar, buscar produtos, ver perfis de fornecedores
- Usuário NÃO pode enviar inquiries (precisa ativação buyer, UC-12) nem listar produtos (precisa upgrade supplier, UC-31/SEQ-17)
- CPF nunca coletado (RN-01.13)

**Rastreabilidade:**
- **UCs:** UC-01
- **RFs:** RF-01.01, RF-01.05
- **RNs:** RN-01.03, RN-01.05, RN-01.13

**Notas de implementação:**
- Rota pública (sem auth middleware)
- Cadastro cria APENAS `profiles` — registros em `buyers`/`suppliers` são criados pelos fluxos de ativação (UC-12) e upgrade (UC-31/SEQ-17)
- Role do usuário é derivado da existência de registros nas tabelas `buyers`/`suppliers` (RN-01.10)
- Para se tornar buyer: precisa enviar primeira inquiry (UC-12, ativação Nível 2)
- Para se tornar supplier: precisa fazer upgrade (UC-31, SEQ-17)

### 3.2. SEQ-02: Login e Autenticação

**Cenário:** Usuário (fornecedor ou comprador) faz login com email/senha. O sistema valida credenciais via Supabase Auth, verifica status da conta (confirmada, ativa), emite JWT + refresh token. Inclui fluxo de refresh token para sessões expiradas.

**Pré-condições:**
- Conta existente no sistema
- Email confirmado (para acesso completo)

**Participantes:**

| Componente | Domínio | Estereótipo |
|-----------|---------|-------------|
| rateLimitMiddleware | Cross-cutting | `<<middleware>>` |
| authService | Identity | `<<service>>` |
| Supabase Auth | External | `<<external>>` |

**Diagrama:**

```mermaid
sequenceDiagram
    autonumber
    participant B as Browser
    participant RL as rateLimitMW
    participant API as API Login
    participant ZV as zodValidation
    participant AS as authService
    participant SA as Supabase Auth

    B->>API: POST /api/auth/login
    Note right of B: email, password
    API->>RL: check(IP, tier: login, 5/min)
    break Rate limit excedido
        RL-->>B: 429 Too Many Requests
        Note over RL: RNF-04.07 — Login: 5 req/min
    end

    API->>ZV: validate(loginSchema)
    API->>AS: login(email, password)
    AS->>SA: signInWithPassword(email, password)

    alt Credenciais inválidas
        SA-->>AS: auth error
        AS-->>B: 401 Email ou senha incorretos
        Note over AS: Mensagem genérica (segurança)
    else Conta não confirmada
        SA-->>AS: email_not_confirmed
        AS-->>B: 403 Confirme seu email primeiro
        Note over AS: RN-01.03 — Reenviar link
    else Conta suspensa
        SA-->>AS: user_banned
        AS-->>B: 403 Conta suspensa
    else Sucesso
        SA-->>AS: session (JWT + refresh_token)
    end

    AS-->>B: 200 access_token, refresh_token, user
    Note over B: Armazenar tokens (httpOnly cookie)

    opt Token expirado (request subsequente)
        B->>API: Request com JWT expirado
        API->>RL: check(IP)
        API->>AS: refreshSession(refresh_token)
        AS->>SA: refreshSession(token)
        alt Refresh token válido
            SA-->>AS: nova session
            AS-->>B: 200 novo access_token
        else Refresh token expirado
            SA-->>AS: invalid_token
            AS-->>B: 401 Sessão expirada, faça login
        end
    end
```

**Fluxos alternativos:**

| # | Condição | Resultado | Referência |
|---|----------|-----------|-----------|
| ALT-1 | Rate limit excedido (5/min) | 429 — aguardar cooldown | RNF-04.07 |
| ALT-2 | Credenciais inválidas | 401 — mensagem genérica | Segurança |
| ALT-3 | Email não confirmado | 403 — oferecer reenvio | RN-01.03 |
| ALT-4 | Conta suspensa | 403 — contato suporte | RN-07.05 |
| ALT-5 | Refresh token expirado | 401 — redirect para login | — |

**Pós-condições:**
- JWT emitido (validade curta, ~1h)
- Refresh token armazenado (validade longa, ~7d)
- Sessão ativa no Supabase Auth

**Rastreabilidade:**
- **UCs:** UC-17
- **RFs:** RF-01.09, RF-01.10
- **RNs:** RN-01.03
- **RNFs:** RNF-04.01, RNF-04.02, RNF-04.04, RNF-04.07

**Notas de implementação:**
- Supabase Auth gerencia bcrypt (cost 10+), JWT e refresh token nativamente
- Rate limit por IP no tier `login` (5 req/min). Após 5 falhas → delay 30s; após 10 → block 15min
- Nunca revelar se o email existe ou não (mensagem genérica "Email ou senha incorretos")
- Tokens via httpOnly cookie (não localStorage) para proteção contra XSS

### 3.3. SEQ-03: Edição de Perfil com Recálculo de Completude

**Cenário:** Fornecedor atualiza dados do perfil (nome fantasia, descrição, logo, endereço, categorias, etc.). O sistema valida, persiste, recalcula a completude conforme RN-02.01, e o novo score impacta o ranking de busca.

**Pré-condições:**
- Fornecedor autenticado com email confirmado
- Perfil existente em `suppliers`

**Participantes:**

| Componente | Domínio | Estereótipo |
|-----------|---------|-------------|
| authMiddleware | Cross-cutting | `<<middleware>>` |
| supplierService | Identity | `<<service>>` |
| supplierRepository | Identity | `<<repository>>` |
| Cloudflare R2 | External | `<<external>>` |

**Diagrama:**

```mermaid
sequenceDiagram
    autonumber
    participant B as Browser
    participant AM as authMiddleware
    participant API as API Supplier
    participant ZV as zodValidation
    participant SS as supplierService
    participant SR as supplierRepo
    participant R2 as Cloudflare R2

    B->>API: PATCH /api/suppliers/{id}
    Note right of B: tradeName, description,<br/>logoFile, address, phone, ...
    API->>AM: validateJWT + checkOwnership(id)
    break Não autenticado ou não é dono
        AM-->>B: 401/403
    end

    API->>ZV: validate(updateSupplierProfileSchema)
    break Validação Zod falha
        ZV-->>B: 400 campos inválidos
    end

    API->>SS: updateProfile(id, data)

    opt Upload de logo incluído
        SS->>R2: uploadImage(file, suppliers/id/logo)
        Note over R2: Resize + WebP conversion
        R2-->>SS: CDN URL
        SS->>SS: data.logoUrl = cdnUrl
    end

    SS->>SR: update(id, data)
    SR-->>SS: supplier atualizado

    SS->>SS: calculateProfileCompleteness(id)
    Note over SS: RN-02.01 — 9 fatores ponderados:<br/>Logo 10%, Desc 15%, End 10%,<br/>Tel 10%, Cat 10%, 3+Prod 20%,<br/>Foto 15%, Horário 5%, Ano 5%
    SS->>SR: updateCompleteness(id, newScore)

    SS-->>B: 200 supplier, completeness
    Note over B: Ranking de busca recalculado<br/>automaticamente (score é fator)

    opt Completude < 50% e não recebeu nudge recente
        Note over SS: RN-02.02 — Agendar nudge<br/>3d, 7d, 14d, 30d+ mensal
    end
```

**Fluxos alternativos:**

| # | Condição | Resultado | RN |
|---|----------|-----------|-----|
| ALT-1 | JWT inválido/expirado | 401 → refresh ou login | — |
| ALT-2 | Supplier ID diferente do JWT | 403 — sem permissão | RLS |
| ALT-3 | Validação Zod falha | 400 — campos detalhados | — |
| ALT-4 | Upload de imagem falha (formato/tamanho) | 422 — aceitar apenas JPG/PNG/WebP, max 5MB | RF-02.03 |

**Pós-condições:**
- Dados do perfil atualizados em `suppliers`
- `profile_completeness` recalculado e persistido
- Logo no Cloudflare R2 (se enviada)
- Ranking de busca reflete novo score de completude

**Rastreabilidade:**
- **UCs:** UC-02
- **RFs:** RF-02.01, RF-02.02, RF-02.03, RF-02.04
- **RNs:** RN-02.01, RN-02.02

**Notas de implementação:**
- Partial update: apenas campos enviados são atualizados (spread + merge)
- Logo upload: processar resize server-side antes de enviar ao R2 (max 500x500px, WebP)
- Completude é calculated e stored (não computed on-read) por ser fator de ranking (performance)
- Invalidar cache ISR da página pública do fornecedor após update

### 4.1. SEQ-04: Cadastro de Produto com Imagens

**Cenário:** Fornecedor cria um produto com nome, descrição, categoria, fotos. O sistema valida, faz upload das imagens para R2, persiste o produto, atualiza o FTS (search_vector), recalcula completude do perfil e dispara revalidação ISR da página SEO.

**Pré-condições:**
- Fornecedor autenticado com email confirmado
- Categoria válida existente no sistema

**Participantes:**

| Componente | Domínio | Estereótipo |
|-----------|---------|-------------|
| authMiddleware | Cross-cutting | `<<middleware>>` |
| productService | Catalog | `<<service>>` |
| productRepository | Catalog | `<<repository>>` |
| supplierService | Identity | `<<service>>` |
| Cloudflare R2 | External | `<<external>>` |
| PostgreSQL | External | `<<external>>` |

**Diagrama:**

```mermaid
sequenceDiagram
    autonumber
    participant B as Browser
    participant AM as authMiddleware
    participant API as API Products
    participant ZV as zodValidation
    participant PS as productService
    participant PR as productRepo
    participant R2 as Cloudflare R2
    participant SS as supplierService
    participant PG as PostgreSQL

    B->>API: POST /api/products
    Note right of B: name, description, categoryId,<br/>images[], unit, minQty, priceRange
    API->>AM: validateJWT + role: supplier
    break Não autenticado
        AM-->>B: 401
    end

    API->>ZV: validate(createProductSchema)
    break Validação falha
        ZV-->>B: 400 campos obrigatórios
        Note over ZV: name, description, categoryId requeridos
    end

    API->>PS: createProduct(supplierId, data)
    PS->>PS: generateSlug(name)
    PS->>PS: generateTags(name, description)
    Note over PS: RN-02.07 — Auto-tags extraídos<br/>de nome + descrição + categoria

    PS->>PR: findBySlugAndSupplier(slug, supplierId)
    opt Slug duplicado
        PS->>PS: slug = slug + contador
    end

    opt Imagens incluídas (max 5)
        loop Para cada imagem
            PS->>R2: uploadImage(file, products/id/)
            Note over R2: Resize + WebP + CDN
            R2-->>PS: imageUrl
        end
    end

    PS->>PR: create(product)
    PR->>PG: INSERT products + product_images
    PG->>PG: trigger atualiza search_vector (tsvector)
    Note over PG: GIN index para FTS

    PS->>SS: calculateProfileCompleteness(supplierId)
    Note over SS: RN-02.01 — 3+ produtos = 20%

    PS-->>B: 201 product
    Note over B: Página ISR revalidada<br/>automaticamente para SEO
```

**Fluxos alternativos:**

| # | Condição | Resultado | RN |
|---|----------|-----------|-----|
| ALT-1 | Campos obrigatórios faltando | 400 — name, description, categoryId | RF-03.01 |
| ALT-2 | Imagem formato inválido | 422 — apenas JPG/PNG/WebP, max 5MB | RF-03.01 |
| ALT-3 | Mais de 5 imagens | 422 — limite de 5 fotos por produto | RF-03.01 |
| ALT-4 | Slug duplicado (mesmo nome + supplier) | Auto-incremento no slug | — |
| ALT-5 | Categoria inexistente | 400 — categoryId inválido | RN-02.04 |

**Pós-condições:**
- Produto criado em `products` com status: active
- Imagens no Cloudflare R2 com URLs CDN
- `search_vector` atualizado via trigger PostgreSQL
- `profile_completeness` do fornecedor recalculado
- Página SEO `/produto/[slug]` será gerada via ISR no próximo acesso

**Rastreabilidade:**
- **UCs:** UC-03
- **RFs:** RF-03.01, RF-03.02, RF-03.03, RF-03.05
- **RNs:** RN-02.03, RN-02.04, RN-02.07

**Notas de implementação:**
- Upload de imagens em paralelo (`Promise.all`) para performance
- Trigger PostgreSQL para `search_vector`: `tsvector_update_trigger` em name + description + tags
- Após INSERT: `revalidatePath('/produto/' + slug)` para ISR
- Sem limite de produtos por fornecedor (RN-02.03 — ilimitado para todos os planos)

### 4.2. SEQ-05: Geração de Página SEO (ISR)

**Cenário:** Usuário ou Googlebot acessa uma página de produto. Cloudflare CDN verifica cache; se miss, Next.js renderiza via ISR com dados do banco, gera meta tags e structured data. Páginas com menos de 3 resultados recebem `noindex`.

**Pré-condições:**
- Produto/fornecedor/categoria existe e está ativo
- Rota pública (sem autenticação)

**Participantes:**

| Componente | Domínio | Estereótipo |
|-----------|---------|-------------|
| Cloudflare CDN | External | `<<external>>` |
| seoService | Search | `<<service>>` |
| productRepository | Catalog | `<<repository>>` |
| supplierRepository | Identity | `<<repository>>` |
| categoryRepository | Catalog | `<<repository>>` |

**Diagrama:**

```mermaid
sequenceDiagram
    autonumber
    participant U as User/Googlebot
    participant CDN as Cloudflare CDN
    participant NJ as Next.js ISR
    participant SeoS as seoService
    participant PR as productRepo
    participant SR as supplierRepo
    participant CR as categoryRepo

    U->>CDN: GET /produto/parafuso-sextavado-sp

    alt Cache HIT (stale-while-revalidate)
        CDN-->>U: HTML cacheado
        Note over CDN: Revalidate: 60s (produtos)<br/>300s (categorias)
    else Cache MISS
        CDN->>NJ: forward request
        NJ->>SeoS: getProductPage(slug)
        SeoS->>PR: findBySlug(slug)

        alt Produto não encontrado
            PR-->>SeoS: null
            SeoS-->>U: 404 Not Found
        else Produto pausado/deletado
            PR-->>SeoS: product (status != active)
            SeoS-->>U: 410 Gone
        else Produto ativo
            PR-->>SeoS: product
        end

        SeoS->>SR: getPublicProfile(product.supplierId)
        SR-->>SeoS: supplier public data
```

## 12. Pendências e Observações

### 12.1 Edge Cases Identificados

| # | Edge Case | SEQ | Decisão Necessária |
|---|-----------|-----|-------------------|
| EC-01 | Comprador altera dados de contato entre envio da inquiry e desbloqueio pelo fornecedor | SEQ-11 | Snapshot no momento da inquiry garante consistência (já decidido, DM-02 do 2.5) |
| EC-02 | Fornecedor é suspenso enquanto tem inquiries pendentes de distribuição | SEQ-09 | Remover das próximas rodadas, mas manter as já distribuídas |
| EC-03 | Stripe webhook chega fora de ordem (payment_failed antes de checkout.completed) | SEQ-13 | Verificar status atual da subscription antes de processar |
| EC-04 | Race condition: dois requests simultâneos de unlock no mesmo inquiry | SEQ-11 | Constraint UNIQUE (inquiry_id, supplier_id) em leads + transação DB |
| EC-05 | Crédito semanal expira no meio de um desbloqueio (domingo 00:01 durante request) | SEQ-11/12 | Transação atômica garante consistência; se expirou, cai no fluxo "sem créditos" |

### 12.2 Cenários Excluídos com Justificativa

| Cenário | Motivo |
|---------|--------|
| CRUD de produtos (edit/pause/delete) | Operação direta: form → validate → update/soft-delete. Sem cross-domain. |
| Ativação de buyer (Nível 2) | Usuário Nível 1 envia inquiry → sistema cria registro `buyers` → Nível 2 ativo. Fluxo direto. |
| Favoritos (toggle) | INSERT/DELETE atômico em `favorites`. Sem lógica complexa. |
| Admin CRUD (fornecedores, categorias, produtos) | Operações admin sem interação cross-domain significativa. |
| Dashboard de métricas | Queries de leitura agregadas. Não gera sequência de interação. |
| Bulk import [VAL] | Job de processamento de arquivo. Fluxo linear sem branches complexos. |

### 12.3 Referência a Pendências (REFERENCIA §17)

- **ORM:** Prisma vs Drizzle (CTO decide) — afeta como repositories são implementados em todos os SEQs
- **Backend stack:** Node.js vs Python/FastAPI (CTO decide) — se Python, API routes viram endpoints FastAPI
- **CNPJ API:** BrasilAPI vs ReceitaWS (avaliar) — afeta SEQ-15, SEQ-17, SEQ-18
- **Convenção lib/:** Domain vs layer (CTO-01 do 3.1) — afeta organização dos services/repos

### 12.4 Observações Técnicas

- **Transações DB:** SEQ-11 (desbloqueio) é o cenário mais crítico para atomicidade. Supabase suporta transações via `rpc()` ou client-side transaction wrapper.
- **Webhooks Stripe:** Devem ser processados de forma idempotente. Manter tabela de `processed_webhook_ids` para deduplicação.
- **Rate limiting:** Implementado no `rateLimitMiddleware` (Cross-cutting, 3.1 seção 3.9). Tiers: navegação 100/min, forms 10/min, login 5/min.
- **Dois algoritmos independentes:** Busca (RN-03.01, SEQ-06) e distribuição (RN-05.04, SEQ-09) são implementações SEPARADAS. Nunca compartilham código de ranking.
