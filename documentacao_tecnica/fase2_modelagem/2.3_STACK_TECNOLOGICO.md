## Convenções deste documento

Cada tecnologia segue o formato:

> **Tecnologia** — categoria (front-end, back-end, infra, etc.)
> **Versão adotada:** versão estável atual
> **Justificativa:** por que essa tecnologia
> **Alternativas descartadas:** o que foi avaliado e por que não entrou
> **Requisitos atendidos:** quais RNFs e RFs essa escolha atende diretamente
> **Custo:** pricing no momento da decisão
> **Risco:** o que pode dar errado e como mitigar

**Princípios que guiaram as escolhas:**

1. **Custo zero no MVP.** Startup pré-receita, sem investimento externo. Toda tecnologia precisa ter free tier funcional ou ser coberta pelos créditos cloud já aprovados (AWS US$1.000, Azure US$1.000, Google Cloud US$300).
2. **SEO desde o dia 1.** SEO programático é o motor de aquisição de compradores. Qualquer escolha que comprometa SSR/SSG, Core Web Vitals ou velocidade de indexação é vetada.
3. **Time de 1 dev.** Vitor (CTO) é o único desenvolvedor. A stack precisa maximizar produtividade de um full-stack solo, com ecossistema maduro e documentação abundante.
4. **Escala sem reescrever.** A arquitetura precisa suportar de 500 a 30.000 fornecedores sem migração de tecnologia (RNF-02.01).
5. **Ecossistema integrado.** Menos ferramentas, mais integração nativa. Reduz overhead operacional e pontos de falha.

## 1. TypeScript — Linguagem principal

**Versão adotada:** 6.0 (lançada em março/2026)
**Categoria:** Linguagem de programação

### Justificativa

TypeScript é a linguagem padrão do ecossistema Next.js/React e permite que front-end e back-end compartilhem tipos, interfaces e validações. Para um time de 1 dev, manter uma única linguagem em toda a stack reduz drasticamente a carga cognitiva e elimina a necessidade de traduzir modelos entre camadas.

O modo strict do TypeScript funciona como uma primeira camada de testes: detecta erros em tempo de compilação que só apareceriam em produção com JavaScript puro. Isso atende diretamente ao RNF-10.01 (tipagem estática strict mode) e permite que o CI rejeite código com type errors antes do merge.

A versão 6.0 é a última baseada em JavaScript. O TypeScript 7.0 (reescrito em Go, até 10x mais rápido) já está em preview e será compatível. Não há risco de obsolescência.

### Alternativas descartadas

| Alternativa | Por que não |
|---|---|
| JavaScript puro | Sem tipagem estática. Mais bugs em runtime, refatoração arriscada. Inadequado para projeto que vai escalar (RNF-10.01) |
| Python (back-end separado) | Quebraria a stack unificada. Dois runtimes, duas cadeias de ferramentas, dois deploys. Overhead inaceitável para 1 dev. Ver seção "Decisão: back-end unificado" |

### Requisitos atendidos

RNF-10.01 (TypeScript strict mode), RNF-10.02 (linting e type checking no CI), RNF-10.05 (qualidade de código via PR reviews).

### Custo

R$0 — open source (licença MIT).

## 2. Next.js — Framework full-stack

**Versão adotada:** 16.2 (lançada em 18/03/2026)
**Categoria:** Framework front-end + back-end (full-stack)

### Justificativa

Next.js é o framework React mais adotado do mundo e a escolha natural para um marketplace que depende de SEO. Suas capacidades de renderização (SSR, SSG, ISR) permitem que cada página de produto, categoria e localidade seja entregue como HTML completo no primeiro request, exatamente o que o Googlebot precisa para indexar (RNF-08.01).

**Por que Next.js resolve problemas específicos da GiroB2B:**

**SEO programático nativo.** O sistema de rotas dinâmicas (`/produto/[slug]`, `/fornecedor-de/[categoria]-em-[cidade]`) gera páginas estáticas em build time (SSG) com revalidação incremental (ISR). Isso significa que com 10.000 produtos, Next.js gera 10.000+ páginas indexáveis automaticamente. O ISR permite que quando um fornecedor edita um produto, a página seja regenerada em até 60 segundos sem rebuildar todo o site (RNF-02.04).

**Core Web Vitals otimizados.** Next.js inclui otimizações automáticas: Image component com lazy loading e formatos modernos (WebP/AVIF), Font optimization que elimina layout shift, Script optimization com estratégias de carregamento. Isso atende diretamente aos RNFs de performance (RNF-01.04 a RNF-01.08).

**API Routes como back-end.** As API Routes do Next.js (Route Handlers) permitem construir toda a API REST dentro do mesmo projeto, eliminando a necessidade de um servidor back-end separado no MVP. Simplifica o deploy (tudo no Vercel) e reduz latência (front e back no mesmo edge).

**Turbopack.** O novo bundler do Next.js 16+ oferece builds até 5x mais rápidos e hot reload em microsegundos. Para um dev solo iterando rápido, a diferença na experiência de desenvolvimento é significativa.

**Versão 16.2 específica:** builds ~400% mais rápidos no dev server, rendering ~50% mais rápido, desserialização de Server Components até 350% mais rápida, 200+ bug fixes.

### Alternativas descartadas

| Alternativa | Por que não |
|---|---|
| Remix | Bom para SPAs interativas, mas SSG e ISR são inferiores ao Next.js. Para um marketplace com milhares de páginas estáticas indexáveis, SSG/ISR é inegociável |
| Nuxt.js (Vue) | Ecossistema menor que React/Next. Menos bibliotecas de componentes, menos profissionais no mercado brasileiro. Vue é excelente, mas para 1 dev que vai precisar contratar, React tem pool de talentos 3x maior |
| Astro | Ótimo para sites de conteúdo estático, mas fraco para interatividade (painéis de fornecedor, CRM, formulários complexos). Teria que adicionar React como "island" e perderia a integração nativa |
| SvelteKit | Framework excelente, mas ecossistema imaturo para B2B. Poucas bibliotecas de componentes corporativos, comunidade menor, mais difícil contratar |
| Back-end separado (FastAPI/Express) | Adicionaria um segundo deploy, um segundo runtime, e latência de rede entre front e back. Para o MVP com 1 dev, o monolito Next.js com API Routes é mais produtivo e mais barato |

### Requisitos atendidos

RNF-01.01 a RNF-01.08 (performance e Core Web Vitals), RNF-02.01 (escalabilidade de 500 a 30K), RNF-02.04 (ISR), RNF-07.02 (responsivo), RNF-07.03 (PWA), RNF-08.01 a RNF-08.07 (SEO técnico completo), RF-05.01 a RF-05.09 (SEO programático).

### Custo

R$0 — open source (licença MIT).

### Risco

**Vendor lock-in com Vercel.** Next.js é open source, mas funcionalidades avançadas (middleware, ISR, Image optimization) funcionam melhor no Vercel. Mitigação: o Build Adapters API do Next.js 16 facilita deploy em outros providers (AWS, Cloudflare). Além disso, os créditos AWS e Azure permitem migração se necessário.

## 3. Tailwind CSS — Estilização

**Versão adotada:** 4.2 (estável, março/2026)
**Categoria:** Framework CSS (utility-first)

### Justificativa

Tailwind elimina a necessidade de escrever CSS customizado para a maioria dos componentes. Para um dev solo construindo dezenas de páginas (produto, categoria, busca, perfil, painéis, admin), a velocidade de prototipação é incomparável. O HTML descreve a interface completa sem alternar entre arquivos.

**Tailwind v4 específico:** engine até 5x mais rápida que v3, builds incrementais em microsegundos, configuração via CSS puro (sem arquivo JS de config), variáveis CSS nativas para temas.

O resultado final é CSS mínimo em produção: o Tailwind faz purge automático e só inclui as classes usadas. Isso contribui diretamente para o RNF-01.07 (peso da página < 500KB).

### Alternativas descartadas

| Alternativa | Por que não |
|---|---|
| CSS Modules | Bom para isolamento, mas lento para prototipar. Cada componente precisa de um arquivo .module.css separado. Overhead alto para 1 dev |
| Styled Components / Emotion | CSS-in-JS aumenta o bundle size e impacta Core Web Vitals (RNF-01.04 a RNF-01.06). Em 2026, a tendência é zero-runtime CSS |
| Bootstrap | Classes semânticas geram interfaces genéricas. Difícil customizar sem sobrescrever tudo. O resultado visual é "template Bootstrap" |
| Vanilla CSS | Produtividade baixa, sem design system, inconsistência inevitável em projeto grande |

### Requisitos atendidos

RNF-01.07 (peso da página), RNF-06.03 (contraste configurável via design tokens), RNF-07.02 (responsivo com breakpoints sm/md/lg/xl nativos).

### Custo

R$0 — open source (licença MIT).

## 4. Node.js — Runtime back-end

**Versão adotada:** 24 LTS (estável, março/2026)
**Categoria:** Runtime JavaScript/TypeScript no servidor

### Justificativa

Node.js é o runtime nativo do Next.js. Usar Node.js no back-end mantém a stack 100% TypeScript, desde o front-end até as queries do banco de dados. Isso é particularmente crítico para:

**Compartilhamento de tipos.** Interfaces TypeScript definidas uma vez (ex: `Supplier`, `Product`, `Inquiry`) são usadas no front-end, na API, e na validação de dados. Zero duplicação, zero divergência.

**API Routes do Next.js.** Rodam nativamente em Node.js. Não há necessidade de um servidor separado no MVP.

**Ecossistema npm.** O maior registro de pacotes do mundo (2.1M+ pacotes). Bibliotecas maduras para tudo que a GiroB2B precisa: validação (zod), ORM (Prisma/Drizzle), email (Resend SDK), pagamentos (Stripe SDK), CNPJ (validação local).

**Node.js 24 LTS:** suporte garantido até abril/2028. Performance melhorada do V8 engine, melhor suporte a ESM, e estabilidade comprovada em produção.

### Decisão: back-end unificado (Node.js) vs. separado (Python/FastAPI)

Esta é uma das pendências registradas (seção 17 da REFERENCIA_CONSOLIDADA.md, item 4): "Stack backend: Node.js ou Python/FastAPI — Vitor decide."

**Recomendação técnica deste documento: Node.js unificado com Next.js API Routes.**

| Critério | Node.js (unificado) | Python/FastAPI (separado) |
|---|---|---|
| Produtividade (1 dev) | Uma linguagem, um deploy, um repo | Duas linguagens, dois deploys, dois repos |
| Custo de infra | R$0 (API Routes no Vercel free) | +R$25-50/mês (Railway para FastAPI) |
| Latência | Zero (front e API no mesmo edge) | +50-100ms (rede entre Vercel e Railway) |
| Tipos compartilhados | Sim (TypeScript end-to-end) | Não (Pydantic no Python, Zod no TypeScript, manter sincronizado) |
| Ecossistema | npm (2.1M pacotes) | pip (500K pacotes), forte em data science |
| IA/ML futuro | Menos maduro para ML pesado | Excelente (TensorFlow, PyTorch, scikit-learn) |
| Pool de talentos BR | Grande (React/Node é a stack mais popular) | Grande, mas separado (devs Python ≠ devs Node) |
| Migração futura | API Routes podem ser extraídas para microserviço Node | Já é separado, mas com overhead desde o dia 1 |

**Veredicto:** Para o MVP e os primeiros 12 meses, Node.js unificado é a escolha de maior produtividade e menor custo. Se no futuro a GiroB2B precisar de capacidades de ML/IA pesadas (matchmaking inteligente, análise de dados), um microserviço Python pode ser adicionado sem reescrever o core.

**⚠️ Esta recomendação será validada com Vitor (CTO) antes da implementação.** Se Vitor optar por Python/FastAPI, os ajustes necessários estão documentados no RNF-10.01 (trocar ESLint/Prettier por black/ruff/mypy) e o deploy back-end migraria para Railway.

### Alternativas descartadas

| Alternativa | Por que não |
|---|---|
| Python/FastAPI | Ver tabela comparativa acima. Excelente tecnologia, mas o overhead de stack separada não se justifica para 1 dev no MVP |
| Go | Performance excepcional, mas ecossistema web menos maduro que Node. Curva de aprendizado alta para um projeto que precisa sair rápido |
| Java/Spring Boot | Enterprise demais para startup pré-receita. Verbose, lento para iterar, pesado em memória |
| PHP/Laravel | Stack madura e produtiva, mas não integra com o ecossistema Next.js/React. Teria que manter front e back completamente separados |
| Deno / Bun | Runtimes promissores, mas ecossistema de bibliotecas e compatibilidade com Next.js ainda não é 100% estável para produção |

### Requisitos atendidos

RNF-01.02 (tempo de resposta API < 500ms), RNF-02.05 (500 req/s), RNF-10.01 (TypeScript strict), RNF-10.02 (CI/CD com linting e type checking).

### Custo

R$0 — open source (licença MIT). Runtime incluso no Vercel (API Routes) e Railway (se back-end separado).

### Risco

**Performance sob carga.** Node.js é single-threaded. Para operações CPU-intensive (ex: processamento de imagens em massa), pode ser necessário worker threads ou um serviço separado. Mitigação: operações pesadas (resize de imagens) devem ser delegadas para Cloudflare Images ou processadas em background jobs.

## 5. PostgreSQL via Supabase — Banco de dados + BaaS

**Versão adotada:** PostgreSQL 15+ (gerenciado pelo Supabase)
**Categoria:** Banco de dados relacional + Backend-as-a-Service

### Justificativa

PostgreSQL é o banco de dados relacional mais avançado do mundo open source, e o Supabase é a plataforma que o transforma em um BaaS completo. A combinação entrega cinco serviços em um só:

**Banco de dados relacional.** PostgreSQL com suporte nativo a JSON, full-text search (`tsvector` + `pg_trgm`), extensões geoespaciais (PostGIS), e Row Level Security (RLS). Isso atende desde a busca textual do MVP (RNF-01.03: queries < 100ms, full-text < 300ms) até geolocalização futura.

**Autenticação.** Supabase Auth oferece email/senha, magic link, e login social (Google, para fase Validação) com JWT. Cobre RNF-04.01 (hashing bcrypt), RNF-04.02 (tokens com expiração), RNF-04.03 (RBAC via RLS). Não é necessário implementar auth do zero.

**Storage.** Object storage com CDN para imagens de produtos, logos de fornecedores, e documentos. Atende RNF-11.03 (nunca armazenar uploads no filesystem). Suporte nativo a transformação de imagens (resize, crop) via URL.

**Realtime.** Subscriptions em tempo real para notificações (inquiry recebida, inquiry visualizada). Elimina a necessidade de WebSocket server separado no MVP.

**Edge Functions.** Serverless functions (Deno runtime) para background jobs: envio de emails, validação de CNPJ, geração de sitemap. Alternativa ao Railway para tarefas que não precisam de servidor dedicado.

### Supabase Free Tier (abril/2026)

| Recurso | Limite gratuito |
|---|---|
| Projetos | 2 |
| Database | 500 MB |
| Auth | 50.000 MAUs |
| Storage | 1 GB |
| Edge Functions | 500.000 invocações/mês |
| Bandwidth (database) | 5 GB egress |
| Bandwidth (storage) | 5 GB egress |
| **Pausa automática** | **Após 7 dias de inatividade** |

**Análise para a GiroB2B:**

500 MB de database é suficiente para ~50.000 produtos com metadados (estimativa: ~10KB/produto). Para os primeiros 12 meses (meta: 10.000 fornecedores, ~30.000 produtos), o free tier cobre.

50.000 MAUs é mais que suficiente. Nos primeiros meses, esperamos centenas, não milhares de usuários ativos.

1 GB de storage é limitante. Com 10.000 produtos e 3 fotos/produto em WebP (~100KB cada), seriam ~3 GB. **Necessário migrar para Supabase Pro ($25/mês) ou usar Cloudflare R2 (10 GB free) para imagens antes de atingir 3.000 produtos.**

**Pausa automática** é o maior risco no free tier. Em produção, o banco precisa estar sempre ativo. Solução: job de keep-alive (cron que faz ping a cada 6 dias) ou upgrade para Pro.

Quando necessário (estimativa: mês 3-4):

|---|---|

### Alternativas descartadas

| Alternativa | Por que não |
|---|---|
| MongoDB Atlas | NoSQL é inadequado para marketplace com relações complexas (fornecedor → produto → inquiry → lead → plano). Joins são essenciais. MongoDB free tier: 512 MB |
| Firebase (Google) | Firestore é NoSQL document-based. Queries complexas são caras e limitadas. Vendor lock-in extremo com Google. Pricing imprevisível em escala |
| PlanetScale (MySQL) | Excelente MySQL serverless, mas removeu free tier em 2024. PostgreSQL tem melhor suporte a full-text search e extensões |
| Neon (PostgreSQL serverless) | Alternativa viável. Free tier generoso (0.5 GB, 190 horas compute). Porém, não inclui auth, storage, ou realtime. Teria que adicionar NextAuth + Cloudflare R2 + Pusher separadamente |
| AWS RDS | PostgreSQL gerenciado, coberto pelos créditos AWS. Mas sem auth, storage, realtime integrados. Setup e manutenção muito mais complexos para 1 dev |

### Requisitos atendidos

RNF-01.03 (queries < 100ms), RNF-02.01 e RNF-02.02 (escalabilidade até 500K produtos), RNF-02.03 (busca nativa com caminho para Meilisearch), RNF-04.01 a RNF-04.03 (auth segura via Supabase Auth), RNF-05.06 (encryption at rest), RNF-11.01 (backups no Pro), RNF-11.03 (storage com CDN).

### Custo

MVP: R$0 (free tier) → R$25/mês (Pro, a partir do mês 3-4). Coberto por créditos cloud.

### Risco

**Vendor dependency.** Supabase é open source e pode ser self-hosted, mas migrar de uma instância gerenciada para self-hosted exige DevOps. Mitigação: o banco é PostgreSQL padrão. Em caso extremo, um pg_dump migra para qualquer PostgreSQL (AWS RDS, Azure, Neon).

**Limites do free tier.** Pausa automática e 500 MB de storage podem forçar upgrade antes do planejado. Mitigação: keep-alive job + Cloudflare R2 para imagens.

## 6. Vercel — Deploy do front-end

**Versão adotada:** Plataforma gerenciada (sem versionamento)
**Categoria:** Plataforma de deploy e hosting

### Justificativa

Vercel é a empresa criadora do Next.js. A integração é nativa e automática: push no GitHub → build → deploy → preview URL. Para um dev solo, eliminar toda a complexidade de infra (nginx, PM2, Docker, load balancer) é crítico.

**O que o Vercel oferece nativamente:**

**Zero-downtime deploy** com rollback instantâneo (RNF-03.03). Cada deploy cria uma nova versão imutável. Reverter = apontar para a versão anterior.

**Edge Network global** com 40+ regiões. Conteúdo estático servido do edge mais próximo do usuário. GRU (São Paulo) é uma das regiões, garantindo latência mínima para o Brasil (RNF-01.01: TTFB < 200ms).

**Preview deployments** automáticos para cada PR. O time (mesmo sendo 1 dev) pode ver a mudança em produção antes do merge. Atende ao fluxo de PR do RNF-10.05.

**Vercel Analytics** embutido (Core Web Vitals reais de campo). Monitora LCP, INP, CLS diretamente dos dados dos usuários, sem setup (RNF-01.04 a RNF-01.06).

**Serverless Functions** para API Routes. Auto-escala de 0 a milhares de requisições sem provisionar servidores.

## 7. Railway — Deploy do back-end (se necessário)

**Versão adotada:** Plataforma gerenciada
**Categoria:** PaaS (Platform-as-a-Service) para serviços de backend

### Justificativa

Se a recomendação de back-end unificado (Next.js API Routes no Vercel) for adotada, Railway será usado apenas para serviços auxiliares: background jobs, workers de processamento, cron jobs que excedam os limites do Vercel.

Se Vitor optar por Python/FastAPI como back-end separado, Railway se torna a plataforma de deploy principal do back-end. Railway oferece deploy via Dockerfile ou Nixpacks (auto-detect), com pricing por uso (cobrado por segundo de CPU/RAM).

## 8. Cloudflare — CDN, segurança e DNS

**Versão adotada:** Plataforma gerenciada
**Categoria:** CDN + WAF + DDoS + DNS

### Justificativa

Cloudflare é a camada de proteção e performance que fica na frente de toda a infraestrutura. O free tier oferece mais do que muitos serviços pagos:

**CDN com bandwidth ilimitado.** Conteúdo estático (imagens, CSS, JS) servido de 330+ data centers globais, incluindo São Paulo. Reduz TTFB e melhora Core Web Vitals sem custo (RNF-01.01, RNF-01.04).

**DDoS protection nativa.** Proteção contra ataques volumétricos na camada de rede. Cloudflare já mitigou ataques de 26M req/s no free tier. Atende RNF-04.10.

**SSL/TLS automático.** Certificados gerenciados, renovação automática, redirect HTTP → HTTPS. Atende RNF-04.08 diretamente.

**DNS autoritativo.** Mais rápido do mundo (resposta em ~11ms). Propagação de mudanças em segundos.

**Cloudflare R2** (object storage): 10 GB grátis, sem egress fees. Alternativa ao Supabase Storage para imagens de produtos, evitando o gargalo de 1 GB do free tier do Supabase.

### Cloudflare Free Tier (abril/2026)

| Recurso | Limite |
|---|---|
| CDN Bandwidth | Ilimitado |
| DDoS Protection | Incluso (L3/L4 + básico L7) |
| SSL/TLS | Incluso (gerenciado) |
| DNS | Incluso |
| Page Rules | 3 |
| R2 Storage | 10 GB + 10M Class B ops/mês |
| Workers | 100K req/dia |
| WAF (avançado) | NÃO incluso (Pro: $20/mês) |

**Nota sobre WAF:** O free tier inclui apenas regras DDoS básicas. O WAF com ruleset OWASP (exigido pelo RNF-04.10) requer o Cloudflare Pro ($20/mês). Para o MVP, a proteção básica combinada com rate limiting na aplicação (RNF-04.07) é suficiente. WAF avançado pode ser adicionado quando houver receita ou com créditos cloud.

### Alternativas descartadas

| Alternativa | Por que não |
|---|---|
| AWS CloudFront | CDN excelente, coberto pelos créditos. Mas setup complexo (certificados ACM, origins, behaviors). Cloudflare faz tudo com 2 cliques |
| Fastly | Muito bom para edge computing, mas sem free tier. Pricing por bandwidth |
| Sem CDN | Inaceitável. Core Web Vitals degradam, SEO sofre, DDoS expõe o servidor. CDN é obrigatório |

### Requisitos atendidos

RNF-01.01 (TTFB via edge), RNF-01.07 (compressão e cache), RNF-04.08 (HTTPS/TLS), RNF-04.09 (security headers), RNF-04.10 (DDoS protection), RNF-08.01 (entrega rápida de HTML), RNF-11.03 (R2 como storage redundante).

### Custo

R$0 (free tier). WAF Pro: $20/mês (quando necessário). Coberto por créditos.

## 9. Resend — Email transacional

**Versão adotada:** API v1
**Categoria:** Serviço de email transacional

### Justificativa

A GiroB2B precisa enviar emails transacionais desde o dia 1: confirmação de cadastro, nova inquiry, inquiry visualizada, perfil incompleto, resumo semanal (RN-09.01, RF-13.01 a RF-13.04). Resend é um serviço moderno, criado para desenvolvedores, com SDK nativo para Node.js/TypeScript.

**Por que Resend e não alternativas:**

**API simples.** Enviar um email é uma chamada de API com 5 linhas. SDK TypeScript com tipos completos. Integração trivial com Next.js API Routes.

**React Email.** Resend mantém o projeto open source React Email, que permite criar templates de email como componentes React. Para a GiroB2B, isso significa que o mesmo dev que faz o front pode fazer os templates de email com a mesma tecnologia.

**Deliverability.** DKIM, SPF, DMARC configurados automaticamente. Domínio personalizado (comercial@girob2b.com.br como remetente).

### Free Tier (abril/2026)

| Recurso | Limite |
|---|---|
| Emails/mês | 3.000 |
| Domínios | 1 |
| Rate limit | 2 emails/segundo |

3.000 emails/mês é suficiente para o MVP (estimativa: 500 fornecedores × 2 emails/mês = 1.000 + compradores + resumos semanais ≈ 2.000/mês). Quando ultrapassar, Pro ($20/mês para 50.000 emails).

### Alternativas descartadas

| Alternativa | Por que não |
|---|---|
| SendGrid (Twilio) | Free tier generoso (100 emails/dia = ~3.000/mês). Porém, API mais antiga, SDK menos ergonômico, deliverability inferior para novos domínios |
| Amazon SES | Mais barato em volume ($0.10/1.000 emails). Mas setup complexo (IAM, SES sandbox, verificação de domínio manual). Pode usar com créditos AWS no futuro para alto volume |
| Mailgun | Bom, mas free tier reduzido para 100 emails/dia com restrições. API menos moderna que Resend |
| Supabase Auth emails | Supabase envia emails de auth (confirmação, reset) nativamente. Mas para emails de negócio (inquiry recebida, resumo semanal), um serviço dedicado é necessário |

### Requisitos atendidos

RF-13.01 a RF-13.04 (notificações), RN-09.01 (eventos de email), RNF-03.04 (degradação graciosa se email falhar).

### Custo

R$0 (free tier). Pro: $20/mês (quando volume crescer). Amazon SES via créditos AWS como alternativa de alto volume.

## 10. Sentry — Error tracking

**Versão adotada:** SaaS gerenciado
**Categoria:** Monitoramento e error tracking

### Justificativa

Sentry captura erros em tempo real, tanto no front-end (React/Next.js) quanto no back-end (API Routes/Node.js). Cada erro inclui stack trace, contexto do usuário, URL, e breadcrumbs de navegação. Para um dev solo que não pode monitorar logs 24/7, alertas automáticos de erros novos são essenciais.

O SDK do Sentry para Next.js integra automaticamente com Server Components, Client Components, API Routes, e middleware. Setup em < 5 minutos.

### Free Tier (abril/2026)

| Recurso | Limite |
|---|---|
| Errors/mês | 5.000 |
| Performance events | 10.000 |
| Replays | 50 |
| Usuários | 1 |
| Retenção | 30 dias |

5.000 errors/mês é mais que suficiente para o MVP. Um sistema saudável deve ter dezenas de erros, não milhares.

### Alternativas descartadas

| Alternativa | Por que não |
|---|---|
| LogRocket | Foco em session replay, não error tracking. Free tier de 1.000 sessions/mês |
| Bugsnag | Bom, mas free tier menor (7.500 events/mês para 1 user) e ecossistema menor |
| Self-hosted Sentry | Possível (open source), mas overhead de manutenção de infraestrutura. Inviável para 1 dev |
| Console.log + logs do Railway | Funciona, mas sem alertas automáticos, sem agrupamento de erros, sem contexto do usuário. Descobrir bugs por acaso não é monitoramento |

### Requisitos atendidos

RNF-09.01 (logging estruturado), RNF-09.02 (error tracking com stack trace e contexto), RNF-09.04 (alertas automáticos).

### Custo

R$0 (free tier).

## 11. PostHog — Product analytics

**Versão adotada:** Cloud gerenciado
**Categoria:** Analytics de produto e comportamento

### Justificativa

A GiroB2B precisa entender o comportamento dos usuários desde o dia 1: quais termos são buscados, quais produtos recebem mais cliques, onde os compradores abandonam o funil, quais fornecedores ativam o perfil completo. PostHog é uma plataforma all-in-one que cobre analytics, funnels, session replays, e feature flags em um único serviço.

**Por que PostHog e não Google Analytics:**

**Privacy-first e LGPD-friendly.** PostHog pode funcionar sem cookies (cookieless tracking), eliminando a necessidade de banner de consentimento para analytics (RNF-05.01). Google Analytics 4 exige consentimento e cookies.

**Free tier generoso.** 1 milhão de eventos/mês, para sempre, sem cartão de crédito. Google Analytics é "gratuito", mas você paga com os dados dos seus usuários.

**Product analytics completo.** Funnels, retention, user paths, feature flags, A/B testing. Tudo que será necessário para otimizar conversão (free → pago) e validar os pesos dos algoritmos de ranking (pendência 7 da REFERENCIA_CONSOLIDADA).

### Free Tier (abril/2026)

| Recurso | Limite |
|---|---|
| Events/mês | 1.000.000 |
| Session recordings | 5.000/mês |
| Feature flags | 1M requests/mês |
| A/B testing | Incluso |
| Funnels, paths, retention | Incluso |

1M eventos/mês é massivamente suficiente para o primeiro ano.

### Alternativas descartadas

| Alternativa | Por que não |
|---|---|
| Google Analytics 4 | "Gratuito" mas exige cookies, banner de consentimento, e envia dados dos usuários para Google. Setup complexo para eventos customizados. Incompatível com postura LGPD-first |
| Plausible | Excelente para métricas simples (pageviews, referrers), mas sem funnels, session replays, ou feature flags. Sem free tier ($9/mês mínimo). Bom complemento, não substituto |
| Mixpanel | Forte em product analytics, mas free tier de apenas 20M events/mês e pricing crescente. Menos integrações que PostHog |
| Amplitude | Similar ao Mixpanel. Free tier de 50K MTUs (monthly tracked users). Pricing complexo |

### Requisitos atendidos

RNF-09.03 (métricas de negócio em tempo real), RNF-09.05 (logging de buscas como ativo estratégico), RNF-05.01 (cookieless tracking, LGPD).

### Custo

R$0 (free tier).

## 14. PWA — Mobile

**Categoria:** Estratégia mobile

### Justificativa

A GiroB2B não terá app nativo no MVP (decisão documentada em 1.7_DEFINICAO_MVP_SCOPE_LOCK.md). Em vez disso, o site será uma PWA (Progressive Web App) instalável, oferecendo experiência próxima de app nativo sem custo de desenvolvimento separado.

Next.js suporta PWA nativamente com configuração de `manifest.json`, service worker (via `next-pwa` ou Workbox), e ícones em múltiplas resoluções.

**O que a PWA oferece:**

Instalável na home screen (Android e iOS). Splash screen customizada. Cache offline de páginas visitadas. Push notifications (fase futura). Funciona como "app" sem App Store.

### Requisitos atendidos

RNF-07.03 (PWA instalável), RNF-07.04 (cache offline).

### Custo

R$0 — nativo do Next.js.
