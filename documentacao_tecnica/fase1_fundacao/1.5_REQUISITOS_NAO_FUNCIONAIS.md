## Convenções deste documento

Cada requisito segue o formato:

> **[RNF-XX.YY]** Descrição do requisito
> **Fase:** MVP | Validação | Monetização | Escala
> **Prioridade:** Alta | Média | Baixa
> **Métrica:** como medir o cumprimento do requisito

Onde:
- **XX** = número da categoria
- **YY** = número do requisito dentro da categoria
- **Fase** indica em qual estágio do roadmap o requisito entra (alinhado com 5W2H, seção WHEN)

**Fases do roadmap:**
- **MVP** (meses 1-3): requisitos essenciais para lançar
- **Validação** (meses 4-6): requisitos para melhorar qualidade e engajamento
- **Monetização** (meses 7-9): requisitos para operação com pagamentos
- **Escala** (meses 10-18): requisitos para crescimento sustentável

## Categoria 01 — Performance

### Tempo de resposta do servidor

**[RNF-01.01]** O tempo de resposta do servidor (TTFB — Time to First Byte) para páginas públicas (produto, categoria, busca, perfil de fornecedor) deve ser inferior a 200ms no p95, considerando usuários no Brasil.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** p95 do TTFB medido via monitoramento sintético (ex: Vercel Analytics, Uptime Robot) < 200ms

**[RNF-01.02]** O tempo de resposta de endpoints de API (autenticação, CRUD de produtos, envio de inquiry) deve ser inferior a 500ms no p95 sob carga normal (até 100 requisições simultâneas).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** p95 de latência dos endpoints < 500ms, medido via APM (Application Performance Monitoring)

**[RNF-01.03]** Consultas ao banco de dados devem ser executadas em menos de 100ms no p95. Consultas que envolvam buscas textuais (full-text search) devem ser executadas em menos de 300ms no p95.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** p95 de query time medido via logs do PostgreSQL ou painel do Supabase

### Core Web Vitals (Google)

Os Core Web Vitals são métricas do Google que impactam diretamente o ranking de SEO. Como o SEO programático é o principal canal de aquisição de compradores, atingir notas "boas" é obrigatório.

**[RNF-01.04]** Todas as páginas públicas devem atingir LCP (Largest Contentful Paint) inferior a 2,5 segundos no p75 dos usuários reais.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** LCP < 2,5s no p75, medido via Google Search Console e/ou Vercel Analytics (dados reais de campo, CrUX)

**[RNF-01.05]** Todas as páginas públicas devem atingir INP (Interaction to Next Paint) inferior a 200ms no p75 dos usuários reais. O INP substituiu o FID como métrica de responsividade em março de 2024.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** INP < 200ms no p75 (dados reais de campo). Contexto: em 2025, ~43% dos websites falhavam nessa métrica (fonte: NitroPack/web.dev, publicado em 2025).

**[RNF-01.06]** Todas as páginas públicas devem atingir CLS (Cumulative Layout Shift) inferior a 0,1 no p75 dos usuários reais.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** CLS < 0,1 no p75 (dados reais de campo)

### Capacidade de carregamento

**[RNF-01.07]** O peso total de cada página pública (HTML + CSS + JS + fontes) não deve exceder 500KB na primeira carga (antes de imagens). Imagens devem ser servidas em formatos modernos (WebP ou AVIF) com lazy loading.
**Fase:** MVP | **Prioridade:** Média
**Métrica:** Page weight medido via Lighthouse ou WebPageTest < 500KB (sem imagens)

**[RNF-01.08]** O tempo total de carregamento de páginas públicas (load completo) deve ser inferior a 3 segundos em conexão 4G (1,6 Mbps de throughput efetivo, que é a simulação padrão do Lighthouse para mobile).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Lighthouse Performance Score >= 90 em modo mobile

## Categoria 02 — Escalabilidade

**[RNF-02.01]** A arquitetura deve suportar crescimento de 500 para 30.000 fornecedores cadastrados sem necessidade de reescrever componentes fundamentais do sistema. O dimensionamento deve ser horizontal (adicionar instâncias) e não vertical (máquina maior).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Teste de carga simulando 30.000 fornecedores, 100.000 produtos e 10.000 buscas/dia sem degradação > 20% no tempo de resposta

**[RNF-02.02]** O banco de dados deve suportar até 500.000 registros de produtos e 1 milhão de inquiries acumuladas sem degradação perceptível de performance nas consultas de busca e listagem.
**Fase:** Validação | **Prioridade:** Alta
**Métrica:** Testes de carga com dataset simulado. Query time de busca < 500ms com 500K produtos.

**[RNF-02.03]** O sistema de busca textual deve ser projetado para migração futura de busca nativa do PostgreSQL (pg_trgm + tsvector) para um serviço dedicado (Meilisearch, Typesense ou Elasticsearch) quando o volume de produtos exceder 100.000 registros.
**Fase:** Escala | **Prioridade:** Média
**Métrica:** Documentar a interface/abstração de busca para permitir troca de provider sem alterar o front-end

**[RNF-02.04]** O sistema de geração de páginas estáticas (SSG) deve suportar regeneração incremental (ISR — Incremental Static Regeneration) para não precisar rebuildar todas as páginas quando um único produto for alterado. Tempo máximo de stale content: 60 segundos para páginas de produto, 5 minutos para páginas de categoria.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** ISR configurado com revalidate adequado por tipo de página

**[RNF-02.05]** O sistema deve suportar pelo menos 500 requisições simultâneas sem erros (HTTP 5xx) ou timeouts, correspondendo a picos de tráfego esperados nos primeiros 12 meses.
**Fase:** Validação | **Prioridade:** Alta
**Métrica:** Teste de carga (k6, Artillery ou similar) com 500 req/s durante 5 minutos, taxa de erro < 0,1%

## Categoria 03 — Disponibilidade e Confiabilidade

**[RNF-03.01]** O sistema deve manter disponibilidade mínima de 99,5% no primeiro ano (equivalente a ~43,8 horas de downtime permitido por ano). Meta para o segundo ano: 99,9% (~8,7 horas/ano).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Uptime medido via serviço externo (UptimeRobot, Better Stack ou similar) com checks a cada 1 minuto

**[RNF-03.02]** O sistema deve implementar health checks automatizados que verifiquem a cada 60 segundos: front-end respondendo (HTTP 200), API respondendo, banco de dados acessível.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Health check endpoint `/api/health` retornando status de todos os componentes críticos

**[RNF-03.03]** O deploy de novas versões deve ser feito com zero-downtime (blue-green deployment ou rolling deployment). O rollback para a versão anterior deve ser possível em menos de 5 minutos.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Zero interrupção perceptível durante deploys. Vercel já oferece isso nativamente para o front-end.

**[RNF-03.04]** O sistema deve degradar graciosamente em caso de falha de serviços externos (API de CNPJ, serviço de email, gateway de pagamento). A falha de um serviço externo não deve derrubar a plataforma inteira.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Teste de falha simulada: desligar cada serviço externo e verificar que o sistema continua operando com funcionalidade reduzida + mensagem informativa ao usuário

**[RNF-03.05]** O sistema deve implementar circuit breaker para chamadas a APIs externas (ReceitaWS, gateway de pagamento, serviço de email), evitando cascata de falhas quando o serviço externo estiver fora do ar.
**Fase:** Validação | **Prioridade:** Média
**Métrica:** Circuit breaker abre após 5 falhas consecutivas; fallback ativado em < 1s

## Categoria 04 — Segurança

### Autenticação e controle de acesso

**[RNF-04.01]** Todas as senhas devem ser armazenadas com hashing seguro (bcrypt com cost factor >= 10, ou Argon2id). Senhas em texto plano nunca devem existir em logs, banco de dados ou transmissão.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Auditoria de código; nenhuma senha em plaintext em qualquer camada

**[RNF-04.02]** Tokens de sessão (JWT ou session tokens) devem ter expiração máxima de 24 horas para sessões ativas e 7 dias para refresh tokens. Tokens devem ser invalidados no logout.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Configuração verificável no código de autenticação

**[RNF-04.03]** O sistema deve implementar controle de acesso baseado em papéis (RBAC): fornecedor, comprador e admin. Um usuário autenticado como comprador não deve conseguir acessar endpoints de fornecedor ou admin, e vice-versa.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Testes automatizados de autorização para cada endpoint protegido

**[RNF-04.04]** Após 5 tentativas de login falhadas consecutivas, o sistema deve aplicar rate limiting progressivo: delay de 30 segundos após 5 tentativas, bloqueio temporário de 15 minutos após 10 tentativas, com notificação por email ao titular da conta.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Teste de brute-force simulado

### Proteção contra ataques (OWASP Top 10 2025)

**[RNF-04.05]** O sistema deve implementar proteção contra as vulnerabilidades do OWASP Top 10 2025, com atenção prioritária a: Broken Access Control (A01), Security Misconfiguration (A02), Injection (A05) e Server-Side Request Forgery (A10).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Scan de segurança automatizado (OWASP ZAP ou similar) sem vulnerabilidades críticas ou altas

**[RNF-04.06]** Todas as entradas de usuário devem ser sanitizadas e validadas no lado do servidor. O sistema deve prevenir SQL Injection (via queries parametrizadas / ORM), XSS (via sanitização de HTML e Content Security Policy) e CSRF (via tokens anti-CSRF ou SameSite cookies).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Nenhuma vulnerabilidade de injection encontrada em testes de penetração

**[RNF-04.07]** O sistema deve implementar rate limiting em todos os endpoints públicos para prevenir abuso: máximo de 100 requisições por minuto por IP para navegação, 10 requisições por minuto por IP para formulários de contato/inquiry, 5 requisições por minuto por IP para login.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Configuração de rate limiting verificável; teste de carga com IPs únicos

### Comunicação e infraestrutura

**[RNF-04.08]** Toda comunicação entre cliente e servidor deve ser criptografada via HTTPS (TLS 1.2 ou superior). Certificado SSL gerenciado via Cloudflare. HTTP deve redirecionar automaticamente para HTTPS.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** SSL Labs score >= A; nenhuma página servida via HTTP puro

**[RNF-04.09]** O sistema deve implementar headers de segurança HTTP: Content-Security-Policy, X-Content-Type-Options (nosniff), X-Frame-Options (DENY ou SAMEORIGIN), Strict-Transport-Security (HSTS com max-age >= 1 ano), Referrer-Policy, Permissions-Policy.
**Fase:** MVP | **Prioridade:** Média
**Métrica:** Verificação via securityheaders.com, nota >= A

**[RNF-04.10]** O sistema deve estar protegido contra ataques DDoS na camada de rede via Cloudflare (já previsto na stack). Regras de firewall (WAF) devem bloquear padrões de ataque conhecidos.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Cloudflare ativo com modo "Under Attack" disponível; WAF com ruleset OWASP habilitado

**[RNF-04.11]** Chaves de API, secrets, credenciais de banco de dados e tokens de serviços externos nunca devem estar no código-fonte. Devem ser gerenciados via variáveis de ambiente (environment variables) nos serviços de deploy (Vercel, Railway).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Nenhum secret encontrado em scan de repositório (ex: git-secrets, trufflehog)

## Categoria 05 — Privacidade e LGPD

A GiroB2B trata dados pessoais de compradores (nome, email, telefone, empresa) e de representantes de fornecedores. O modelo de negócio depende do compartilhamento controlado desses dados. A conformidade com a LGPD (Lei 13.709/2018) é obrigatória e deve ser considerada desde a arquitetura (privacy by design).

**[RNF-05.01]** O sistema deve coletar consentimento explícito, livre, informado e inequívoco do comprador antes de compartilhar seus dados de contato com fornecedores. O texto do consentimento deve ser específico sobre: quais dados serão compartilhados (nome, email, telefone, empresa), com quem (fornecedores com plano pago), para qual finalidade (responder à solicitação de cotação).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Checkbox de consentimento obrigatório no formulário de inquiry, com texto revisado por advogado

**[RNF-05.02]** O sistema deve permitir que qualquer titular de dados (comprador ou representante de fornecedor) exerça seus direitos previstos na LGPD: acesso aos dados, correção, exclusão (direito ao esquecimento), portabilidade e revogação do consentimento. O prazo de atendimento deve ser de até 15 dias corridos.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Endpoint ou interface administrativa para processar solicitações de titulares; log de cada solicitação com prazo de atendimento

**[RNF-05.03]** O sistema deve manter um Registro de Operações de Tratamento de Dados Pessoais (ROPA), documentando: quais dados são coletados, a base legal para cada tratamento, a finalidade, o período de retenção e com quem os dados são compartilhados.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Documento ROPA mantido e atualizado a cada mudança no fluxo de dados

**[RNF-05.04]** Os dados pessoais de compradores que revogarem o consentimento devem ser anonimizados ou excluídos em até 30 dias. Inquiries já desbloqueadas por fornecedores pagantes devem manter apenas dados agregados (categoria da inquiry, cidade), sem dados pessoais.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Script de anonimização testado; dados pessoais não recuperáveis após exclusão

**[RNF-05.05]** O sistema deve exibir Política de Privacidade e Termos de Uso acessíveis em todas as páginas (link no rodapé). Ambos os documentos devem ser redigidos em linguagem clara e revisados por advogado antes do lançamento.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Links funcionais no rodapé; documentos em linguagem acessível

**[RNF-05.06]** Dados pessoais armazenados no banco de dados devem ter criptografia em repouso (encryption at rest). Dados sensíveis em trânsito já são cobertos pelo RNF-04.08 (HTTPS).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Encryption at rest habilitado no PostgreSQL/Supabase (Supabase já oferece isso por padrão em projetos hospedados)

**[RNF-05.07]** O sistema deve definir e aplicar períodos de retenção para dados pessoais: dados de contas inativas há mais de 24 meses devem ser anonimizados automaticamente; logs de acesso devem ser retidos por no máximo 12 meses; inquiries devem ser retidas enquanto houver relação comercial ativa ou por 5 anos após o último contato (prazo prescricional do Código Civil).
**Fase:** Validação | **Prioridade:** Média
**Métrica:** Jobs automatizados de limpeza/anonimização; relatório mensal de dados retidos

**[RNF-05.08]** Em caso de incidente de segurança envolvendo dados pessoais, o sistema deve possibilitar a notificação à ANPD (Autoridade Nacional de Proteção de Dados) e aos titulares afetados em até 72 horas, conforme Art. 48 da LGPD. Para isso, o sistema deve manter logs de acesso a dados pessoais que permitam identificar o escopo de um eventual vazamento.
**Fase:** Validação | **Prioridade:** Alta
**Métrica:** Plano de resposta a incidentes documentado; logs de acesso a dados pessoais habilitados

## Categoria 06 — Acessibilidade

**[RNF-06.01]** Todas as páginas públicas devem atender ao nível AA das WCAG 2.1 (Web Content Accessibility Guidelines). Os quatro princípios devem ser respeitados: Perceivable (perceptível), Operable (operável), Understandable (compreensível), Robust (robusto).
**Fase:** Validação | **Prioridade:** Média
**Métrica:** Score de acessibilidade >= 90 no Lighthouse; auditoria com axe-core sem violações críticas

**[RNF-06.02]** Todas as imagens informativas (logos de fornecedores, fotos de produtos) devem ter texto alternativo (alt text) descritivo. Imagens decorativas devem ter alt vazio (`alt=""`).
**Fase:** MVP | **Prioridade:** Média
**Métrica:** Campo de alt text obrigatório no upload de fotos de produto; auditoria automática

**[RNF-06.03]** O contraste de texto sobre fundo deve atender à razão mínima de 4.5:1 para texto normal e 3:1 para texto grande (>= 18pt ou 14pt bold), conforme WCAG 2.1 critério 1.4.3.
**Fase:** MVP | **Prioridade:** Média
**Métrica:** Verificação automática de contraste no design system (plugin Figma ou teste automatizado)

**[RNF-06.04]** Todos os formulários (cadastro, login, inquiry, busca) devem ser navegáveis por teclado (tab order lógico) e ter labels associados aos campos via atributo `for`/`id`. Mensagens de erro devem ser associadas ao campo correspondente via `aria-describedby`.
**Fase:** MVP | **Prioridade:** Média
**Métrica:** Teste manual de navegação por teclado em todos os formulários; sem "armadilhas de foco"

**[RNF-06.05]** O HTML deve usar elementos semânticos (`<header>`, `<nav>`, `<main>`, `<footer>`, `<article>`, `<section>`) e landmarks ARIA para que leitores de tela consigam interpretar a estrutura da página.
**Fase:** MVP | **Prioridade:** Média
**Métrica:** Auditoria de estrutura semântica via axe-core ou similar

## Categoria 07 — Compatibilidade

### Navegadores

**[RNF-07.01]** O sistema deve funcionar corretamente nos seguintes navegadores (2 últimas versões estáveis): Chrome (desktop e mobile), Safari (desktop e iOS), Firefox (desktop), Edge (desktop), Samsung Internet (mobile). Esses navegadores cobrem ~98% do tráfego web no Brasil.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Testes manuais ou automatizados (Playwright, Cypress) nos navegadores listados

### Dispositivos

**[RNF-07.02]** O sistema deve ser mobile-first e responsivo, funcionando em telas de 320px (smartphones pequenos) a 2560px (monitores widescreen). Breakpoints mínimos: 320px, 640px (sm), 768px (md), 1024px (lg), 1280px (xl).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Teste visual em cada breakpoint; Lighthouse mobile score >= 90

**[RNF-07.03]** O sistema deve funcionar como PWA (Progressive Web App) instalável em dispositivos Android e iOS. Deve incluir: manifest.json, service worker para cache offline de páginas estáticas, ícones em múltiplas resoluções, splash screen.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Lighthouse PWA score >= 90; instalável via "Adicionar à tela inicial"

### Conectividade

**[RNF-07.04]** Páginas já visitadas devem ser acessíveis offline (via service worker cache). Ao detectar perda de conexão, o sistema deve exibir mensagem informativa ao invés de página de erro genérica.
**Fase:** Validação | **Prioridade:** Média
**Métrica:** Teste offline: desconectar internet e verificar que páginas cacheadas continuam acessíveis

## Categoria 08 — SEO Técnico

O SEO programático é o principal canal de aquisição de compradores (custo zero). Os requisitos técnicos de SEO são tão críticos quanto os funcionais.

**[RNF-08.01]** Todas as páginas públicas (produto, categoria, localidade, categoria+localidade, perfil de fornecedor) devem ser renderizadas no servidor (SSR) ou pré-geradas (SSG). O HTML completo deve estar disponível no primeiro request HTTP, sem depender de JavaScript para renderizar o conteúdo principal.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Visualizar source (Ctrl+U) de cada tipo de página e confirmar que o conteúdo principal está no HTML

**[RNF-08.02]** Cada página pública deve ter meta tags únicas e otimizadas geradas dinamicamente: `<title>` (máximo 60 caracteres), `<meta description>` (máximo 155 caracteres), `<link rel="canonical">`, Open Graph tags (og:title, og:description, og:image, og:url) e Twitter Card tags.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Auditoria com Screaming Frog ou similar; nenhuma página com meta tags duplicadas ou ausentes

**[RNF-08.03]** URLs devem ser amigáveis, em português, lowercase, usando hífens como separador. Padrões definidos em RF-05.01 a RF-05.04. Redirecionamentos 301 devem ser implementados para qualquer mudança de URL.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Nenhuma URL com caracteres especiais, underscores ou IDs numéricos expostos

**[RNF-08.04]** O sitemap XML deve ser gerado automaticamente e atualizado diariamente. Deve incluir todas as páginas indexáveis com lastmod, changefreq e priority. Sitemaps grandes devem ser divididos em sub-sitemaps (máximo 50.000 URLs por arquivo).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Sitemap acessível em `/sitemap.xml`; URLs do sitemap correspondendo às páginas reais

**[RNF-08.05]** O sistema deve gerar automaticamente arquivo `robots.txt` bloqueando: painel admin, páginas de login, APIs internas, e permitindo: todas as páginas públicas, sitemap.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** `robots.txt` acessível na raiz; nenhuma página pública bloqueada inadvertidamente

**[RNF-08.06]** Dados estruturados (Schema.org em JSON-LD) devem ser implementados em todas as páginas de produto (`Product`), perfil de fornecedor (`Organization`, `LocalBusiness`), categorias (`ItemList`) e breadcrumbs (`BreadcrumbList`). Os dados estruturados devem ser validados sem erros no Rich Results Test do Google.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Validação via Google Rich Results Test; zero erros, zero avisos críticos

**[RNF-08.07]** Páginas sem conteúdo útil (categorias vazias, localidades sem fornecedores) devem retornar meta tag `noindex` para evitar diluição da qualidade do índice. Quando ganharem conteúdo, o `noindex` deve ser removido automaticamente.
**Fase:** MVP | **Prioridade:** Média
**Métrica:** Páginas com < 3 resultados devem ter `noindex`; verificação via crawl automatizado

## Categoria 09 — Observabilidade

**[RNF-09.01]** O sistema deve implementar logging estruturado (JSON) em todos os serviços, incluindo: timestamp, nível (info, warn, error), request ID, user ID (quando autenticado), endpoint, tempo de resposta e stack trace em caso de erro.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Logs acessíveis via console do serviço de deploy (Vercel Functions logs, Railway logs) ou via serviço de log (Better Stack, Axiom)

**[RNF-09.02]** Erros não tratados (exceptions) devem ser capturados automaticamente e reportados para um serviço de error tracking (Sentry é a opção recomendada por ter plano gratuito generoso). Cada erro deve incluir: stack trace, contexto do usuário, URL, body do request (sem dados sensíveis).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Sentry (ou similar) integrado; alertas configurados para erros novos e recorrentes

**[RNF-09.03]** O sistema deve coletar métricas de negócio em tempo real ou near-real-time: cadastros por dia, inquiries por dia, buscas realizadas, termos mais buscados, páginas mais acessadas, taxa de conversão (visita → inquiry).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Dashboard de analytics (Vercel Analytics + Plausible ou PostHog) com métricas de negócio visíveis

**[RNF-09.04]** Alertas automáticos devem ser configurados para: downtime (sistema indisponível por > 1 minuto), taxa de erro > 5% em janela de 5 minutos, tempo de resposta p95 > 2 segundos, falha em jobs agendados (ex: geração de sitemap).
**Fase:** Validação | **Prioridade:** Alta
**Métrica:** Alertas configurados com notificação via email e/ou Slack/Discord/WhatsApp

**[RNF-09.05]** Todas as buscas realizadas por compradores devem ser logadas (termo buscado, filtros aplicados, número de resultados, cliques nos resultados). Esses dados são ativos estratégicos para entender a demanda do mercado (ver IDEIAS_E_FUTURO.md, seção 4 — "Dados como ativo estratégico").
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Tabela de logs de busca no banco de dados; relatório semanal de termos mais buscados disponível no painel admin

## Categoria 10 — Manutenibilidade e Deploy

**[RNF-10.01]** O código-fonte deve seguir convenções consistentes de formatação (Prettier), linting (ESLint com regras de Next.js) e tipagem estática (TypeScript strict mode). A configuração deve ser versionada no repositório.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** ESLint e Prettier configurados; CI rejeita código com erros de lint ou type errors

**[RNF-10.02]** O sistema deve ter pipeline de CI/CD automatizado: a cada push para a branch principal, o pipeline deve executar linting, type checking, testes automatizados e deploy para ambiente de staging. Deploy para produção deve ser manual (aprovação via PR).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Pipeline configurado no GitHub Actions (ou equivalente); deploys automáticos para staging

**[RNF-10.03]** O sistema deve ter cobertura mínima de testes automatizados de 60% nas funções críticas: autenticação, busca, envio de inquiry, RBAC, validação de CNPJ. Não exigimos 80%+ de cobertura global (irrealista para MVP), mas as rotas críticas devem estar cobertas.
**Fase:** Validação | **Prioridade:** Média
**Métrica:** Relatório de cobertura de testes gerado pelo CI; funções críticas com >= 60% de cobertura

**[RNF-10.04]** Ambientes de desenvolvimento, staging e produção devem ser isolados, com bancos de dados e variáveis de ambiente separados. Dados de produção nunca devem ser usados em ambientes de desenvolvimento sem anonimização.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Três ambientes configurados; verificação de que não há cross-contamination de dados

**[RNF-10.05]** O repositório deve seguir Git Flow simplificado: branch `main` (produção), branch `develop` (staging), feature branches para cada funcionalidade. Pull Requests obrigatórios para merge em `main`, com pelo menos 1 review.
**Fase:** MVP | **Prioridade:** Média
**Métrica:** Branch protection rules configuradas no GitHub

## Categoria 11 — Backup e Recuperação de Desastres

**[RNF-11.01]** O banco de dados PostgreSQL deve ter backups automáticos diários com retenção mínima de 30 dias. O RPO (Recovery Point Objective) máximo é de 24 horas para dados gerais e 1 hora para dados transacionais (inquiries, leads, pagamentos).
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Backups diários configurados no Supabase (que já oferece backups automáticos no plano Pro) ou via pgBackRest; logs de backup verificados semanalmente

**[RNF-11.02]** O RTO (Recovery Time Objective) máximo é de 4 horas: em caso de perda total do banco de dados, o sistema deve ser restaurado e operacional em até 4 horas.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Teste de restauração de backup executado pelo menos 1 vez por trimestre; tempo de restauração documentado

**[RNF-11.03]** Imagens de produtos e logos de fornecedores devem ser armazenadas em serviço de object storage com redundância geográfica (ex: Supabase Storage, AWS S3, Cloudflare R2). Nunca armazenar uploads apenas no filesystem do servidor.
**Fase:** MVP | **Prioridade:** Alta
**Métrica:** Uploads servidos via CDN; nenhum arquivo local no servidor de aplicação

**[RNF-11.04]** A estratégia de backup deve seguir a regra 3-2-1: 3 cópias dos dados, em 2 tipos de mídia diferentes, com 1 cópia offsite (geograficamente separada). Na prática para o MVP: banco no Supabase (cópia 1), backup automático do Supabase (cópia 2), export periódico para outro cloud provider via créditos AWS ou Azure (cópia 3).
**Fase:** Validação | **Prioridade:** Média
**Métrica:** Três cópias verificáveis; teste de restauração da cópia offsite pelo menos 1 vez por semestre

## ⚠️ Observações pendentes

1. **Revisão jurídica da LGPD:** Os requisitos RNF-05.01 a RNF-05.08 foram elaborados com base na lei e em boas práticas, mas o texto do consentimento, a Política de Privacidade e os Termos de Uso devem ser revisados por advogado especializado em proteção de dados antes do lançamento.

2. **Definição final de stack com Vitor (CTO):** Requisitos de manutenibilidade (RNF-10.01 a RNF-10.05) assumem TypeScript + Next.js + GitHub. Se o CTO optar por Python/FastAPI no back-end, os requisitos de linting e CI precisam ser ajustados (ex: black + ruff + mypy ao invés de ESLint + Prettier + TypeScript).

3. **Plano do Supabase:** O RNF-11.01 menciona backups automáticos do Supabase. No plano gratuito, os backups não são configuráveis. No plano Pro ($25/mês), backups diários com retenção de 7 dias são incluídos. Avaliar se os créditos cloud cobrem esse custo nos primeiros meses.

4. **Custos de ferramentas de observabilidade:** Sentry (error tracking), Plausible/PostHog (analytics), UptimeRobot (monitoramento) possuem planos gratuitos suficientes para o MVP. Documentar limites de cada plano gratuito no artefato 4.2 (Custos de Ferramentas/SaaS).
