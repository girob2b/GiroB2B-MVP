## Convenções deste documento

Cada requisito segue o formato:

> **[RF-XX.YY]** Descrição do requisito
> **Fase:** MVP | Validação | Monetização | Escala
> **Prioridade:** Alta | Média | Baixa

Onde:
- **XX** = número do módulo
- **YY** = número do requisito dentro do módulo
- **Fase** indica em qual estágio do roadmap o requisito entra (ver 5W2H, seção WHEN)

**Fases do roadmap:**
- **MVP** (meses 1-3): funcionalidades essenciais para lançar
- **Validação** (meses 4-6): funcionalidades para testar engajamento
- **Monetização** (meses 7-9): funcionalidades para gerar receita
- **Escala** (meses 10-18): funcionalidades para crescer

## Módulo 01 — Cadastro e Autenticação

### Cadastro unificado (Nível 1 — Usuário)

**[RF-01.01]** O sistema deve permitir que qualquer pessoa se cadastre na plataforma informando: email, senha, nome completo, telefone, cidade e estado. Não há escolha de role no cadastro — todo mundo entra como **usuário (Nível 1)**. Sem CNPJ, sem dados de empresa.
**Fase:** MVP | **Prioridade:** Alta

**[RF-01.02]** O sistema deve validar o CNPJ informado em tempo real consultando a base da Receita Federal (via ReceitaWS, BrasilAPI ou similar), verificando se está ativo e retornando razão social, natureza jurídica e situação cadastral. A validação ocorre em dois momentos: upgrade para fornecedor (obrigatório, RF-01.13) e verificação de empresa pelo buyer (opcional, RF-01.14).
**Fase:** MVP | **Prioridade:** Alta

**[RF-01.03]** O sistema deve impedir o uso de CNPJs com situação cadastral diferente de "Ativa" no upgrade para fornecedor (RF-01.13). No caso de verificação de buyer (RF-01.14), CNPJ não-ativo = selo não concedido, mas buyer não é bloqueado.
**Fase:** MVP | **Prioridade:** Alta

**[RF-01.04]** O sistema deve impedir cadastro duplicado do mesmo CNPJ como fornecedor. Caso já exista, oferecer a opção de "reivindicar" o perfil (ver RF-01.12). ⚠️ PENDÊNCIA: comportamento quando buyer informa CNPJ que já existe como supplier (vincular conta? sugerir login? — Vitor decide).
**Fase:** MVP | **Prioridade:** Alta

**[RF-01.05]** O sistema deve enviar email de confirmação após o cadastro (Nível 1). O usuário só acessa funcionalidades autenticadas após confirmar o email.
**Fase:** MVP | **Prioridade:** Alta

### Ativação de buyer (Nível 2 — Comprador)

**[RF-01.06]** O sistema deve ativar o perfil de comprador (Nível 2) quando o usuário tenta enviar uma inquiry pela primeira vez. Nesse momento, o sistema solicita aceite do checkbox LGPD (RF-01.08) e opcionalmente empresa e CNPJ (para selo de verificação, RF-01.14). Um registro é criado na tabela `buyers` e o role do usuário passa a ser derivado como `buyer`.
**Fase:** MVP | **Prioridade:** Alta

**[RF-01.07]** O sistema deve permitir que usuários de Nível 1 (sem ativação de buyer) realizem buscas, naveguem pelo catálogo, vejam perfis públicos de fornecedores e explorem categorias. A ativação de buyer é exigida apenas no momento de enviar uma inquiry.
**Fase:** MVP | **Prioridade:** Alta

**[RF-01.08]** O sistema deve coletar consentimento explícito do comprador (checkbox LGPD) no momento da ativação de buyer (Nível 2), autorizando o compartilhamento de seus dados de contato com fornecedores que possuam plano pago. O texto deve ser claro e específico sobre quais dados serão compartilhados e com quem. Ver RN-01.07. O consentimento NÃO é exigido no cadastro base (Nível 1).
**Fase:** MVP | **Prioridade:** Alta

### Autenticação

**[RF-01.09]** O sistema deve oferecer login por email + senha unificado para todos os perfis (usuário, comprador, fornecedor). O login é o mesmo independentemente do nível de engajamento.
**Fase:** MVP | **Prioridade:** Alta

**[RF-01.10]** O sistema deve oferecer recuperação de senha via email.
**Fase:** MVP | **Prioridade:** Alta

**[RF-01.11]** O sistema deve oferecer login social (Google) como alternativa ao cadastro por email.
**Fase:** Validação | **Prioridade:** Média

### Reivindicação de perfil

**[RF-01.12]** O sistema deve permitir que o representante legal de uma empresa reivindique um perfil pré-cadastrado (criado por scraping ou pelo admin). O processo de reivindicação deve verificar a identidade via email no domínio da empresa ou envio de documento.
**Fase:** Validação | **Prioridade:** Média

### Upgrade para fornecedor (Nível 3 — Supplier)

**[RF-01.13]** O sistema deve permitir que um usuário (Nível 1) ou buyer (Nível 2) faça upgrade para fornecedor (Nível 3) informando: CNPJ (obrigatório, validado via RF-01.02), razão social, e endereço completo (CEP, rua, número, bairro, cidade, estado). Após validação do CNPJ, um registro é criado na tabela `suppliers` e o role passa a incluir `supplier`. Se o usuário já era buyer, mantém ambos os roles (dual-role, RN-01.11).
**Fase:** MVP | **Prioridade:** Alta

### Selo "Empresa Verificada" e filtro

**[RF-01.14]** O sistema deve conceder o selo "Empresa Verificada" a buyers que informarem CNPJ e tiverem validação positiva via BrasilAPI (situação "Ativa"). O selo é exibido nas inquiries enviadas pelo buyer e é visível para o fornecedor **antes** de gastar crédito para desbloquear o lead. Buyer sem CNPJ ou com CNPJ não-ativo aparece sem selo (não é bloqueio). Ver RN-01.06.
**Fase:** MVP | **Prioridade:** Alta

**[RF-01.15]** O sistema deve oferecer ao fornecedor a opção de filtrar inquiries/leads recebidos para exibir apenas os de "Empresas Verificadas". O filtro é uma preferência do fornecedor (não regra global), default **desativado**. Ao ativar, o sistema deve exibir aviso: "Ativar este filtro pode reduzir significativamente o volume de solicitações recebidas." Ver RN-01.12.
**Fase:** MVP | **Prioridade:** Média

### Barra de completude e nudges para buyer

**[RF-01.16]** O sistema deve exibir uma barra de completude para buyers, similar à existente para suppliers (RF-02.04). Cálculo progressivo sugerido: ~40% (dados base: email, senha, nome, telefone, cidade/estado) → ~60% (empresa informada) → ~80% (CNPJ validado, selo concedido) → ~90% (foto/avatar adicionada) → 100% (primeira inquiry enviada). ⚠️ PENDÊNCIA: valores exatos da barra (Vitor valida).
**Fase:** MVP | **Prioridade:** Média

**[RF-01.17]** O sistema deve exibir nudges de completude para buyers em momentos estratégicos: após enviar primeira inquiry, no dashboard do comprador, e em emails de follow-up. Mensagem sugerida: "Compradores com empresa verificada recebem mais propostas." O CNPJ permanece opcional — o nudge incentiva mas nunca obriga. ⚠️ PENDÊNCIA: frequência e canais dos nudges (email? in-app? push? — Vitor define).
**Fase:** MVP | **Prioridade:** Média

## Módulo 02 — Perfil do Fornecedor

**[RF-02.01]** O fornecedor deve poder editar seu perfil com: logo, descrição da empresa (até 2.000 caracteres), endereço completo (opcional exibir rua), horário de funcionamento, site, redes sociais, ano de fundação e número de funcionários (faixas).
**Fase:** MVP | **Prioridade:** Alta

**[RF-02.02]** O fornecedor deve poder selecionar até 5 categorias/setores de atuação a partir de uma árvore hierárquica padronizada (categoria > subcategoria).
**Fase:** MVP | **Prioridade:** Alta

**[RF-02.03]** O fornecedor deve poder adicionar fotos da empresa (até 10 fotos, máximo 5MB cada).
**Fase:** MVP | **Prioridade:** Média

**[RF-02.04]** O sistema deve exibir um indicador de "perfil completo" (barra de progresso) incentivando o fornecedor a preencher todas as informações. Perfis completos devem ter prioridade nos resultados de busca.
**Fase:** MVP | **Prioridade:** Média

**[RF-02.05]** O perfil público do fornecedor deve exibir: nome da empresa, logo, descrição, categorias, localização (cidade/estado), produtos listados, selo de verificação (se aplicável) e botão "Solicitar Cotação".
**Fase:** MVP | **Prioridade:** Alta

**[RF-02.06]** O fornecedor com plano pago deve poder adicionar vídeo institucional ao perfil (embed YouTube/Vimeo).
**Fase:** Monetização | **Prioridade:** Baixa

**[RF-02.07]** O fornecedor com plano Premium deve poder usar um subdomínio personalizado (ex: nomedaempresa.girob2b.com.br).
**Fase:** Escala | **Prioridade:** Baixa

## Módulo 03 — Catálogo de Produtos

**[RF-03.01]** O fornecedor deve poder cadastrar produtos/serviços informando: nome, descrição (até 1.000 caracteres), categoria, subcategoria, fotos (até 5 por produto, máximo 5MB cada), unidade de medida, quantidade mínima de pedido e faixa de preço (opcional).
**Fase:** MVP | **Prioridade:** Alta

**[RF-03.02]** O sistema deve permitir listagem ilimitada de produtos para todos os fornecedores, inclusive os gratuitos.
**Fase:** MVP | **Prioridade:** Alta

**[RF-03.03]** O fornecedor deve poder editar, pausar (ocultar temporariamente) e excluir seus produtos a qualquer momento.
**Fase:** MVP | **Prioridade:** Alta

**[RF-03.04]** O sistema deve categorizar produtos em uma árvore hierárquica padronizada (ex: Embalagens > Plásticas > Sacolas). A árvore de categorias deve ser expansível pelo admin.
**Fase:** MVP | **Prioridade:** Alta

**[RF-03.05]** O sistema deve gerar automaticamente tags/palavras-chave a partir do nome e descrição do produto para melhorar a busca interna e o SEO.
**Fase:** MVP | **Prioridade:** Média

**[RF-03.06]** O sistema deve exibir para cada produto: fotos, descrição, dados do fornecedor (nome, cidade, selo), botão "Solicitar Cotação" e link para o perfil do fornecedor.
**Fase:** MVP | **Prioridade:** Alta

**[RF-03.07]** O sistema deve permitir importação em massa de produtos via planilha (CSV/XLSX) com template padronizado fornecido pela plataforma.
**Fase:** Validação | **Prioridade:** Média

## Módulo 04 — Busca e Descoberta

**[RF-04.01]** O sistema deve oferecer busca textual por nome de produto, categoria, palavras-chave e nome de fornecedor.
**Fase:** MVP | **Prioridade:** Alta

**[RF-04.02]** O sistema deve oferecer filtros de busca por: categoria, subcategoria, cidade, estado, faixa de preço (quando informada) e verificação (selo GiroB2B Verificado).
**Fase:** MVP | **Prioridade:** Alta

**[RF-04.03]** O sistema deve exibir resultados de busca com relevância baseada em: correspondência textual, completude do perfil do fornecedor, localização geográfica e nível do plano (planos pagos com prioridade sobre gratuitos).
**Fase:** MVP | **Prioridade:** Alta

**[RF-04.04]** O sistema deve oferecer navegação por categorias (browse) como alternativa à busca textual: página de categorias principais > subcategorias > lista de produtos.
**Fase:** MVP | **Prioridade:** Alta

**[RF-04.05]** O sistema deve exibir sugestões de autocompletar (autocomplete) durante a digitação no campo de busca, baseadas em categorias, produtos e fornecedores existentes.
**Fase:** Validação | **Prioridade:** Média

**[RF-04.06]** O sistema deve exibir na página inicial: categorias em destaque, produtos recentes, fornecedores verificados e termos populares de busca.
**Fase:** MVP | **Prioridade:** Média

**[RF-04.07]** O comprador deve poder salvar fornecedores como "favoritos" para acesso rápido posterior (requer cadastro).
**Fase:** Validação | **Prioridade:** Baixa

**[RF-04.08]** O sistema deve permitir que compradores comparem até 3 fornecedores lado a lado (tabela comparativa com dados do perfil e produtos).
**Fase:** Escala | **Prioridade:** Baixa

## Módulo 05 — SEO Programático

**[RF-05.01]** O sistema deve gerar automaticamente uma página individual para cada produto listado, com URL amigável no formato `/produto/[slug-do-produto]-[cidade]` (ex: `/produto/embalagens-plasticas-sao-paulo`).
**Fase:** MVP | **Prioridade:** Alta

**[RF-05.02]** O sistema deve gerar automaticamente páginas de categoria com URL amigável no formato `/categoria/[slug]` (ex: `/categoria/embalagens`), listando todos os produtos daquela categoria.
**Fase:** MVP | **Prioridade:** Alta

**[RF-05.03]** O sistema deve gerar automaticamente páginas de localidade no formato `/fornecedores/[cidade]-[estado]` (ex: `/fornecedores/sao-paulo-sp`), listando fornecedores daquela região.
**Fase:** MVP | **Prioridade:** Alta

**[RF-05.04]** O sistema deve gerar automaticamente páginas combinadas de categoria + localidade no formato `/fornecedor-de/[categoria]-em-[cidade]` (ex: `/fornecedor-de/embalagens-em-sao-paulo`).
**Fase:** MVP | **Prioridade:** Alta

**[RF-05.05]** Todas as páginas geradas automaticamente devem incluir meta tags otimizadas (title, description, Open Graph, canonical URL) com base nos dados do produto/fornecedor/categoria.
**Fase:** MVP | **Prioridade:** Alta

**[RF-05.06]** O sistema deve gerar e atualizar automaticamente um sitemap XML com todas as páginas indexáveis, submetendo-o ao Google Search Console.
**Fase:** MVP | **Prioridade:** Alta

**[RF-05.07]** O sistema deve utilizar renderização server-side (SSR) ou geração estática (SSG) para todas as páginas públicas, garantindo que o conteúdo esteja disponível para crawlers sem necessidade de JavaScript.
**Fase:** MVP | **Prioridade:** Alta

**[RF-05.08]** O sistema deve implementar dados estruturados (Schema.org) nas páginas de produto e fornecedor, utilizando os tipos `Product`, `Organization` e `LocalBusiness` conforme aplicável.
**Fase:** MVP | **Prioridade:** Média

**[RF-05.09]** O sistema deve implementar breadcrumbs navegáveis em todas as páginas internas (Home > Categoria > Subcategoria > Produto).
**Fase:** MVP | **Prioridade:** Média

## Módulo 06 — Inquiries (Solicitações de Cotação / RFQs)

**[RF-06.01]** O comprador deve poder enviar uma inquiry (solicitação de cotação) para um fornecedor específico a partir da página do produto ou do perfil do fornecedor. O formulário deve conter: descrição do que precisa, quantidade estimada, prazo desejado e dados de contato (nome, empresa, email, telefone).
**Fase:** MVP | **Prioridade:** Alta

**[RF-06.02]** O sistema deve enviar notificação por email ao fornecedor quando ele receber uma nova inquiry.
**Fase:** MVP | **Prioridade:** Alta

**[RF-06.03]** O fornecedor deve poder visualizar no seu painel todas as inquiries recebidas, com status (nova, visualizada, respondida, arquivada).
**Fase:** MVP | **Prioridade:** Alta

**[RF-06.04]** Para fornecedores no plano gratuito, a inquiry deve exibir apenas a descrição do pedido e a cidade do comprador, ocultando nome, empresa, email e telefone. Uma mensagem deve indicar que os dados de contato estão disponíveis nos planos pagos. O selo "Empresa Verificada" (RF-01.14) é visível independentemente do plano — ajuda o fornecedor a qualificar o lead antes da decisão de assinatura.
**Fase:** MVP | **Prioridade:** Alta

**[RF-06.05]** Para fornecedores com plano pago e créditos disponíveis, a inquiry deve exibir todos os dados de contato do comprador. O acesso a esses dados consome 1 crédito de lead.
**Fase:** Monetização | **Prioridade:** Alta

**[RF-06.06]** O comprador deve poder enviar uma inquiry genérica (sem direcionamento a um fornecedor específico) do tipo "Procuro fornecedor de [produto] em [cidade]". O sistema distribui essa inquiry para fornecedores relevantes da categoria e região.
**Fase:** Validação | **Prioridade:** Média

**[RF-06.07]** O sistema deve limitar o número de inquiries que um comprador pode enviar por dia (ex: 10/dia) para evitar spam.
**Fase:** MVP | **Prioridade:** Média

**[RF-06.08]** O sistema deve permitir que o fornecedor denuncie inquiries falsas ou spam. Inquiries denunciadas por múltiplos fornecedores são revisadas pelo admin.
**Fase:** Validação | **Prioridade:** Média

## Módulo 07 — Sistema de Leads e Créditos

> **Nota:** Este módulo entra na fase de Monetização (meses 7-9). No MVP, inquiries são exibidas sem sistema de créditos.

**[RF-07.01]** O sistema deve gerenciar créditos de lead por fornecedor. Cada plano concede uma quantidade semanal de créditos (Starter: 5, Pro: 15, Premium: 30+). Créditos não utilizados na semana expiram (não acumulam).
**Fase:** Monetização | **Prioridade:** Alta

**[RF-07.02]** O fornecedor deve poder "desbloquear" os dados de contato de um comprador (inquiry) consumindo 1 crédito de lead. O desbloqueio é irreversível.
**Fase:** Monetização | **Prioridade:** Alta

**[RF-07.03]** O sistema deve exibir para o fornecedor: créditos restantes na semana, histórico de créditos usados, leads desbloqueados e data de renovação dos créditos.
**Fase:** Monetização | **Prioridade:** Alta

**[RF-07.04]** O sistema deve oferecer pacotes avulsos de créditos extras para fornecedores que esgotem sua cota semanal, compráveis sem necessidade de upgrade de plano.
**Fase:** Monetização | **Prioridade:** Média

**[RF-07.05]** O sistema deve distribuir inquiries genéricas (RF-06.06) para múltiplos fornecedores da categoria/região, limitando a no máximo 5 fornecedores por inquiry para manter o valor do lead.
**Fase:** Monetização | **Prioridade:** Alta

**[RF-07.06]** Na distribuição de inquiries genéricas, o sistema deve priorizar fornecedores por: nível do plano (Premium > Pro > Starter), relevância da categoria, proximidade geográfica e tempo de resposta histórico.
**Fase:** Monetização | **Prioridade:** Alta

## Módulo 08 — Planos e Assinaturas

> **Nota:** Este módulo entra na fase de Monetização (meses 7-9).

**[RF-08.01]** O sistema deve oferecer três planos de assinatura para fornecedores: Giro Starter (R$79/mês ou R$790/ano), Giro Pro (R$199/mês ou R$1.990/ano), Giro Premium (R$399/mês ou R$3.990/ano).
**Fase:** Monetização | **Prioridade:** Alta

**[RF-08.02]** O fornecedor deve poder assinar, fazer upgrade, fazer downgrade e cancelar seu plano pelo painel. Mudanças de plano entram em vigor no próximo ciclo de cobrança.
**Fase:** Monetização | **Prioridade:** Alta

**[RF-08.03]** O sistema deve integrar com gateway de pagamento (Stripe, Mercado Pago ou PagSeguro) para cobranças recorrentes via cartão de crédito, boleto e PIX.
**Fase:** Monetização | **Prioridade:** Alta

**[RF-08.04]** O sistema deve aplicar automaticamente os benefícios do plano ao perfil do fornecedor: créditos semanais, selo Verificado (Pro e Premium), posicionamento na busca, subdomínio personalizado (Premium).
**Fase:** Monetização | **Prioridade:** Alta

**[RF-08.05]** O sistema deve enviar notificações de cobrança: lembrete 3 dias antes da renovação, confirmação de pagamento, aviso de falha na cobrança (com 3 tentativas antes de suspender).
**Fase:** Monetização | **Prioridade:** Alta

**[RF-08.06]** O sistema deve oferecer período de trial gratuito de 7 dias no plano Starter para novos fornecedores, sem necessidade de cartão de crédito.
**Fase:** Monetização | **Prioridade:** Média

## Módulo 09 — Painel do Fornecedor (Dashboard)

**[RF-09.01]** O fornecedor deve ter acesso a um painel com visão geral: inquiries recebidas (total e novas), produtos listados, visualizações do perfil e completude do perfil.
**Fase:** MVP | **Prioridade:** Alta

**[RF-09.02]** O painel deve exibir lista de inquiries recebidas com filtros por status (nova, visualizada, respondida, arquivada) e por data.
**Fase:** MVP | **Prioridade:** Alta

**[RF-09.03]** O painel deve permitir gerenciamento completo de produtos: listar, criar, editar, pausar, excluir.
**Fase:** MVP | **Prioridade:** Alta

**[RF-09.04]** O painel deve exibir analytics básicos: visualizações do perfil por período, inquiries recebidas por período, produtos mais visualizados.
**Fase:** Validação | **Prioridade:** Média

**[RF-09.05]** Para fornecedores pagantes, o painel deve exibir CRM de leads: lista de leads desbloqueados com dados de contato, status de acompanhamento (novo, em negociação, fechado, perdido), notas e histórico de interação.
**Fase:** Monetização | **Prioridade:** Alta

**[RF-09.06]** Para fornecedores pagantes, o painel deve exibir relatório mensal de performance: leads recebidos, leads desbloqueados, taxa de resposta, comparação com mês anterior.
**Fase:** Monetização | **Prioridade:** Média

## Módulo 10 — Painel do Comprador

**[RF-10.01]** O comprador deve ter acesso a um painel simples com: inquiries enviadas (com status), fornecedores salvos como favoritos e dados do perfil.
**Fase:** MVP | **Prioridade:** Média

**[RF-10.02]** O comprador deve poder ver o histórico de suas inquiries e o status de cada uma (enviada, visualizada pelo fornecedor).
**Fase:** Validação | **Prioridade:** Média

**[RF-10.03]** O comprador deve poder configurar alertas para ser notificado quando novos fornecedores se cadastrarem em categorias de seu interesse.
**Fase:** Validação | **Prioridade:** Baixa

## Módulo 11 — Verificação de Fornecedores

**[RF-11.01]** O sistema deve realizar verificação automática básica de todo fornecedor cadastrado: CNPJ ativo, situação cadastral regular. Resultado exibido como indicador no perfil (ex: "CNPJ verificado").
**Fase:** MVP | **Prioridade:** Alta

**[RF-11.02]** O sistema deve oferecer selo "GiroB2B Verificado" para fornecedores dos planos Pro e Premium, que inclui: verificação de CNPJ + confirmação de endereço + validação de identidade do representante.
**Fase:** Monetização | **Prioridade:** Alta

**[RF-11.03]** O selo de verificação deve ser exibido no perfil do fornecedor, nos resultados de busca e nas páginas de produto.
**Fase:** Monetização | **Prioridade:** Alta

**[RF-11.04]** O sistema deve permitir que compradores denunciem fornecedores por informações falsas, produtos inexistentes ou comportamento inadequado. Denúncias geram review do admin.
**Fase:** MVP | **Prioridade:** Média

## Módulo 12 — Administração (Back-office)

**[RF-12.01]** O admin deve ter acesso a um painel com métricas gerais: total de fornecedores, compradores, produtos, inquiries (por período), planos ativos, MRR.
**Fase:** MVP | **Prioridade:** Alta

**[RF-12.02]** O admin deve poder gerenciar fornecedores: visualizar, editar, suspender, excluir contas. Incluindo moderação de perfis e produtos.
**Fase:** MVP | **Prioridade:** Alta

**[RF-12.03]** O admin deve poder gerenciar a árvore de categorias: criar, editar, reorganizar, mesclar e desativar categorias e subcategorias.
**Fase:** MVP | **Prioridade:** Alta

**[RF-12.04]** O admin deve poder moderar produtos: aprovar, rejeitar ou solicitar edição de listagens que violem os termos de uso.
**Fase:** MVP | **Prioridade:** Média

**[RF-12.05]** O admin deve poder visualizar e tratar denúncias de compradores e fornecedores.
**Fase:** MVP | **Prioridade:** Média

**[RF-12.06]** O admin deve poder criar perfis pré-cadastrados de fornecedores (via scraping ou inserção manual) que podem ser reivindicados posteriormente.
**Fase:** Validação | **Prioridade:** Média

**[RF-12.07]** O admin deve ter acesso a relatórios analíticos: termos mais buscados, categorias com mais demanda, regiões com mais atividade, funil de conversão (cadastro > perfil completo > inquiry > lead pago).
**Fase:** Validação | **Prioridade:** Média

## Módulo 13 — Notificações

**[RF-13.01]** O sistema deve enviar notificações por email para: confirmação de cadastro, nova inquiry recebida (fornecedor), inquiry visualizada (comprador), lembrete de perfil incompleto, resumo semanal de atividade.
**Fase:** MVP | **Prioridade:** Alta

**[RF-13.02]** O sistema deve enviar notificações push (PWA) para fornecedores quando receberem novas inquiries.
**Fase:** Validação | **Prioridade:** Média

**[RF-13.03]** O sistema deve enviar notificações via WhatsApp Business API para fornecedores pagantes quando receberem novas inquiries (opt-in).
**Fase:** Escala | **Prioridade:** Média

**[RF-13.04]** O fornecedor deve poder configurar suas preferências de notificação: quais tipos receber e por qual canal (email, push, WhatsApp).
**Fase:** Validação | **Prioridade:** Média

## Módulo 14 — Páginas Institucionais e Conteúdo

**[RF-14.01]** O sistema deve oferecer páginas institucionais: Sobre, Como Funciona (fornecedor), Como Funciona (comprador), Planos e Preços, Contato, Termos de Uso, Política de Privacidade.
**Fase:** MVP | **Prioridade:** Alta

**[RF-14.02]** O sistema deve oferecer uma seção de blog/conteúdo integrada para publicação de artigos sobre setores, dicas de compra B2B e guias para fornecedores.
**Fase:** Validação | **Prioridade:** Média

**[RF-14.03]** O sistema deve exibir na página inicial um campo de busca proeminente, categorias em destaque e CTAs claros: "Cadastre-se" (Nível 1, para todos) e "Quero Vender" (upgrade para fornecedor, Nível 3).
**Fase:** MVP | **Prioridade:** Alta

## ⚠️ Observações pendentes

1. **Verticais industriais do Márcio:** Levantar com o Márcio quais subsetores industriais ele tem mais acesso (metalúrgica, plásticos, química, alimentícia, etc.). Essa informação pode influenciar a priorização da árvore de categorias (RF-03.04, RF-12.03) e as páginas SEO programáticas (Módulo 05).

2. **Algoritmo de distribuição de leads:** O detalhamento do algoritmo de priorização (RF-07.05 e RF-07.06) será aprofundado no artefato 1.6 (Regras de Negócio). Ver IDEIAS_E_FUTURO.md, seção 2, para os fatores a considerar.

3. **LGPD e compartilhamento de dados:** O consentimento do comprador (RF-01.08) precisa ser revisado por advogado antes da implementação. Ver IDEIAS_E_FUTURO.md, seção 4 (questão regulatória).

4. **SLA de leads:** Definir o que acontece quando o fornecedor pagante não recebe o número mínimo de leads prometido pelo plano (garantia de satisfação, mês grátis, etc.). Ver IDEIAS_E_FUTURO.md, seção 4 (retenção e churn).
