# Definição do MVP — Scope Lock — GiroB2B

**Versão:** 1.0
**Data:** 02/04/2026
**Autor:** Gustavo (CEO) + Claude (Arquiteto)
**Público:** Time de desenvolvimento + investidores/aceleradoras

---

## Objetivo deste documento

Este documento é um **contrato de escopo**. Ele define, com precisão, o que entra e o que fica de fora do MVP (Produto Mínimo Viável) da GiroB2B. Toda funcionalidade, regra ou requisito técnico que não esteja listado aqui como "dentro do MVP" está automaticamente fora do escopo inicial.

O propósito é evitar scope creep, alinhar expectativas do time de desenvolvimento e dar clareza para investidores e aceleradoras sobre o que será entregue nos primeiros 3 meses de desenvolvimento.

**Regra de ouro:** se uma funcionalidade não é necessária para que um fornecedor cadastre produtos e um comprador envie uma inquiry, ela não entra no MVP.

---

## Definição do MVP em uma frase

> Um marketplace web com **cadastro unificado e progressão por engajamento** (user → buyer → supplier), onde fornecedores fazem upgrade com validação de CNPJ e listam produtos gratuitamente, compradores encontram fornecedores via busca e SEO programático, e enviam inquiries com dados de contato visíveis apenas para fornecedores pagantes. Compradores com CNPJ verificado recebem selo "Empresa Verificada".

---

## Cronograma e meta

**Prazo de desenvolvimento:** 3 meses (meses 1-3 do roadmap)
**Meta de lançamento:** fornecedores conseguem cadastrar perfil e produtos; compradores conseguem buscar, navegar por categorias e enviar inquiries; páginas são indexadas pelo Google.
**Meta de supply:** 500 a 1.000 fornecedores cadastrados nos 3 primeiros meses (aquisição via Márcio em campo + onboarding manual).

## O que ENTRA no MVP

| Ref. | Funcionalidade | Justificativa |
|---|---|---|
| RF-01.01, 01.05 | **Cadastro unificado (Nível 1):** email, senha, nome, telefone, cidade/estado. Sem CNPJ, sem role. Confirmação por email. | Reduzir fricção de entrada — ver valor antes de se comprometer |
| RF-01.06 a 01.08 | **Ativação de buyer (Nível 2):** trigger = enviar inquiry. Aceitar LGPD. CNPJ opcional (para selo). | Sem compradores não existem inquiries |
| RF-01.13 | **Upgrade para fornecedor (Nível 3):** CNPJ obrigatório + razão social + endereço completo. Validação via BrasilAPI. | Sem fornecedores não existe plataforma |
| RF-01.02 a 01.04 | Validação de CNPJ em tempo real, bloqueio de CNPJ inativo, impedimento de duplicatas | Integridade dos dados desde o dia 1 |
| RF-01.09, 01.10 | Login unificado email+senha e recuperação de senha | Funcionalidade básica de autenticação |
| RN-01.01 a 01.03 | CNPJ ativo para supplier, sem duplicatas, perfil visível após confirmar email | Integridade dos dados desde o dia 1 |
| RN-01.05 a 01.07 | Usuário Nível 1 navega sem ser buyer, LGPD na ativação de buyer (Nível 2) | Menor fricção possível |
| RN-01.10, 01.11 | Role derivado (user/buyer/supplier/dual), dual-role permitido | Flexibilidade — donos de empresa são buyer E supplier |
| RF-01.14 | **Selo "Empresa Verificada"** para buyers com CNPJ validado. Visível ao supplier antes do desbloqueio. | Qualificar leads sem obrigar CNPJ |
| RF-01.15 | **Filtro "Apenas Empresas Verificadas"** — preferência do supplier, default OFF, com aviso | Ferramenta de qualificação para o fornecedor |
| RF-01.16 | **Barra de completude do buyer** — progressão visual do perfil | Engajamento e dados mais completos |
| RF-01.17 | **Nudges de completude** — incentivos para completar perfil/CNPJ | Incentiva sem obrigar |
| RN-01.12, 01.13 | Filtro default OFF com aviso; CPF nunca coletado | Decisões explícitas de negócio |

**Fora do MVP (deliberadamente):**

| Funcionalidade | Por que fica de fora | Quando entra |
|---|---|---|
| Login social (Google) — RF-01.11 | Email+senha é suficiente para validar o modelo. Login social reduz fricção mas não é essencial | Validação (mês 4-6) |

| Múltiplos usuários por conta — RN-01.04 | Complexidade desnecessária; no MVP, 1 conta = 1 usuário | Escala (mês 10-18) |

### 6. Inquiries (Solicitações de Cotação)

| Ref. | Funcionalidade | Justificativa |
|---|---|---|
| RF-06.01 | Inquiry direcionada: comprador envia para fornecedor específico | Fluxo principal de conversão |
| RF-06.02 | Notificação por email ao fornecedor | Fornecedor precisa saber que recebeu uma inquiry |
| RF-06.03 | Painel com lista de inquiries e status | Gestão básica das solicitações |
| RF-06.04 | Inquiry com dados ocultos para fornecedor gratuito (exibe descrição, quantidade, prazo e cidade; oculta nome, empresa, email, telefone) | Isca para conversão futura: "assine para ver o contato" |
| RF-06.07 | Limite de 10 inquiries/dia por comprador | Anti-spam |
| RN-04.01, 04.02 | Regras de envio: limite diário, inquiry direcionada é exclusiva e formulário estruturado | Integridade do sistema |
| RN-04.04 | Deduplicação de inquiries (mesmo comprador + fornecedor + produto em 48h) | Anti-spam inteligente |
| RN-04.05 | Dados visíveis vs. ocultos por tipo de plano | Mecânica de monetização já funcional (mesmo antes de cobrar) |
| RN-04.08 | Ciclo de vida da inquiry: Nova → Visualizada → Respondida/Arquivada/Denunciada | Gestão organizada |

**Fora do MVP:**

| Funcionalidade | Por que fica de fora | Quando entra |
|---|---|---|
| Inquiry genérica (distribuída para múltiplos fornecedores) — RF-06.06, RN-04.03 | Requer algoritmo de distribuição por rodadas (complexo); no MVP, comprador envia para fornecedor específico | Validação (mês 4-6) |
| Desbloqueio de dados com crédito — RF-06.05, RN-04.06 | Requer sistema de créditos e pagamento | Monetização (mês 7-9) |
| Denúncia de inquiries — RF-06.08 | Volume baixo no início; moderação manual resolve | Validação (mês 4-6) |
| Lembrete 48h e arquivamento 7d — RN-04.09 | Automação de ciclo de vida; no MVP, status é manual | Validação (mês 4-6) |

**Nota importante sobre inquiries no MVP:** No MVP, o fornecedor gratuito vê a inquiry com dados ocultos e a mensagem "Assine para ver o contato". Porém, como não existe sistema de pagamento no MVP, nenhum fornecedor consegue de fato assinar. A mecânica serve para (a) treinar o comprador no fluxo, (b) coletar dados de demanda, (c) preparar o funil de conversão para a fase de Monetização. Nos bastidores, a equipe pode liberar manualmente dados de contato para fornecedores estratégicos nos primeiros meses (onboarding white-glove).

## Módulos INTEIROS fora do MVP

Estes módulos não têm nenhum requisito no MVP. São sistemas completos que entram em fases posteriores.

### Sistema de Leads e Créditos (Módulo 07)
**Todos os 6 requisitos (RF-07.01 a RF-07.06) ficam de fora.**
Entra na fase de Monetização (meses 7-9). No MVP, inquiries funcionam sem sistema de créditos. Fornecedores veem dados ocultos com mensagem de "assine para acessar", mas não existe mecanismo de assinatura. A mecânica é educativa e de coleta de dados.

### Planos e Assinaturas (Módulo 08)
**Todos os 6 requisitos (RF-08.01 a RF-08.06) ficam de fora.**
Entra na fase de Monetização (meses 7-9). No MVP, não existe cobrança, gateway de pagamento, trial ou gestão de planos. A prioridade é provar que fornecedores e compradores usam a plataforma antes de cobrar.

### Algoritmo de Distribuição de Leads (Grupo 05 das Regras de Negócio)
**Todas as 10 regras (RN-05.01 a RN-05.10) ficam de fora.**
Entra na fase de Monetização (meses 7-9). No MVP, só existem inquiries direcionadas (comprador → fornecedor específico). O modelo de rodadas por tempo (Premium → Pro → Starter) só faz sentido com planos pagos.

### Planos e Assinaturas (Grupo 06 das Regras de Negócio)
**Todas as 10 regras (RN-06.01 a RN-06.10) ficam de fora.**
Mesma justificativa: não há monetização no MVP.

### SLA de Leads (Grupo 08 das Regras de Negócio)
**RN-08.01 e RN-08.02 ficam de fora** (Monetização). RN-08.03 é meta interna para a fase de Validação.

## Critérios de "pronto" do MVP

O MVP é considerado pronto para lançamento quando **todos** os critérios abaixo forem atendidos:

### Funcional

- [ ] Usuário consegue se cadastrar (Nível 1: email, senha, nome, telefone, cidade/estado) e confirmar email
- [ ] Usuário/buyer consegue fazer upgrade para fornecedor (Nível 3) com CNPJ validado + endereço completo
- [ ] Fornecedor consegue completar perfil (logo, descrição, categorias)
- [ ] Fornecedor consegue cadastrar, editar, pausar e excluir produtos
- [ ] Comprador consegue buscar por texto e navegar por categorias
- [ ] Comprador consegue aplicar filtros (categoria, cidade, estado)
- [ ] Comprador consegue enviar inquiry direcionada para um fornecedor (ativação de buyer Nível 2 no primeiro envio)
- [ ] Buyer pode informar CNPJ e receber selo "Empresa Verificada"
- [ ] Supplier pode ativar filtro "Apenas Empresas Verificadas" no painel
- [ ] Buyer vê barra de completude do perfil
- [ ] Fornecedor consegue visualizar inquiries no painel (com dados ocultos)

## Riscos e premissas do MVP

### Premissas

1. **Fornecedores se cadastram sem incentivo financeiro.** A proposta de "aparecer no Google gratuitamente" é suficiente para atrair 500+ fornecedores nos 3 primeiros meses via abordagem presencial do Márcio.

2. **SEO programático gera tráfego orgânico em 3-6 meses.** Baseado em benchmarks: páginas com conteúdo original, Schema.org e boa performance técnica começam a ranquear em 2-4 meses para keywords de cauda longa com baixa competição ("fornecedor de embalagens em Campinas").

3. **Compradores enviam inquiries mesmo sem garantia de resposta.** O custo para o comprador é zero (preencher um formulário). Em marketplaces B2B, a taxa de conversão visitante→inquiry fica entre 2% e 5% (IndiaMART reporta ~2,6% dos suppliers como pagantes, indicando que o volume de inquiries é muito superior).

4. **Free tiers de infraestrutura são suficientes para o MVP.** Vercel (100GB bandwidth), Supabase (500MB database, 50K auth users), Railway (500h/mês). Créditos AWS ($1.000) e Azure ($1.000) como buffer.

### Riscos

| Risco | Impacto | Mitigação |
|---|---|---|
| Márcio não consegue cadastrar 500 fornecedores em 3 meses | Alto — sem supply, sem demand | Meta ajustável; começar com verticais mais acessíveis (embalagens, autopeças); onboarding assistido (equipe cadastra pelo fornecedor) |
| SEO demora mais que 6 meses para gerar tráfego | Médio — atrasa validação | Conteúdo complementar (LinkedIn, blog externo); distribuição manual de inquiries nos primeiros meses |
| Fornecedores não preenchem perfil completo | Médio — qualidade baixa do catálogo | Barra de progresso + lembretes + onboarding ativo (Márcio ajuda a preencher presencialmente) |
| Comprador envia inquiry e ninguém responde (fornecedor não vê dados) | Alto — mata a demanda | Liberação manual de dados para fornecedores estratégicos nos primeiros meses (white-glove) |
| LGPD: consentimento insuficiente ou mal implementado | Alto — risco legal | Consultar advogado antes do lançamento (pendência herdada do RF 1.4) |

## Política de mudança de escopo

Qualquer adição ao escopo do MVP deve passar por esta análise:

1. **É necessário para o fluxo fornecedor-cadastra → comprador-busca → comprador-envia-inquiry?** Se não, fica de fora.
2. **Quanto tempo de desenvolvimento adiciona?** Se mais de 1 semana, fica de fora.
3. **Existe alternativa manual?** Se sim (ex: moderar manualmente em vez de filtros automáticos), usar a alternativa manual e automatizar depois.
4. **A decisão é reversível?** Se implementar agora não impede de mudar depois, pode entrar. Se cria debt técnico irreversível, fica de fora.

Toda mudança de escopo deve ser documentada neste arquivo como adendo, com data e justificativa.
